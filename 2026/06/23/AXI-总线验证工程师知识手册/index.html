第一部分：AXI 协议基础架构

第1章 五通道与握手机制

1.1 核心理论

五个独立通道：

  • 读地址通道（AR）：发送读请求地址和控制信号
  • 读数据通道（R）：返回读数据和响应
  • 写地址通道（AW）：发送写请求地址和控制信号
  • 写数据通道（W）：发送写数据
  • 写响应通道（B）：返回写响应

通道分离的目的是流水线化——地址可以在前一笔数据未完成时提前发出。

VALID/READY 双向握手：

规则 说明
VALID由发送方控制 一旦拉高，必须保持到握手成功
READY由接收方控制 可随时拉高或拉低
握手成功条件 VALID && READY 同时为高
黄金法则 VALID不能依赖于READY（否则死锁）

1.2 完整例题

【例题1.1：握手时序分析】

某AXI总线上，VALID在T1周期上升沿拉高，READY在T3周期上升沿拉高。请问数据在哪个周期完成传输？如果READY一直不拉高，VALID应该保持多久？

解析：

  • 握手发生在T3周期（VALID && READY同时为高）
  • 如果READY一直不拉高，VALID必须无限期保持，直到握手完成
  • 验证时，需要构造READY延迟多个周期的场景，检查DUT是否能正确处理

【例题1.2：死锁场景识别】

下面这段RTL逻辑是否存在死锁风险？为什么？

解析：

  • 存在死锁风险。逻辑等价于：只有在ready为低时才拉高valid，一旦ready变高就拉低valid
  • 如果ready初始为低，valid拉高，但握手条件是valid&&ready，此时ready为低，无法握手
  • 而valid拉高后，下一拍如果ready仍为低，valid继续为高；但这里valid在ready变高前不会变低
  • 更严重的问题是：如果这段逻辑在组合路径中，valid依赖于ready的当前值，形成组合环
  • 正确做法：valid应该是独立的状态机输出，不依赖ready的当前值

第2章 突发传输（Burst）核心计算

2.1 核心理论

三个核心参数：

参数 含义 计算公式
SIZE 每拍传输字节数 字节数 = 2^SIZE
LENGTH 传输拍数 实际拍数 = LENGTH + 1
总字节数 一次突发总数据量 总字节 = 2^SIZE × (LENGTH + 1)

三种BURST类型：

类型 地址计算 典型用途
INCR 每拍地址 += 2^SIZE 普通内存访问
WRAP 递增到上边界后回绕 Cache line填充
FIXED 所有拍地址相同 FIFO访问

4KB边界规则：

  • 任何突发不能跨越4KB（0x1000）地址边界
  • 如果起始地址+总字节超过4KB边界，Interconnect必须拆分

2.2 完整例题

【例题2.1：基础突发计算】

Master发起一笔写事务：

  • 起始地址：0x2000_0000
  • SIZE：3
  • LENGTH：4

问：每拍传输多少字节？共几拍？总字节多少？

解析：

  • 每拍字节 = 2^3 = 8字节
  • 总拍数 = 4 + 1 = 5拍
  • 总字节 = 8 × 5 = 40字节

【例题2.2：地址序列计算（INCR）】

接上题，写出每拍的地址。

解析：

  • 拍0：0x2000_0000
  • 拍1：0x2000_0000 + 8 = 0x2000_0008
  • 拍2：0x2000_0000 + 16 = 0x2000_0010
  • 拍3：0x2000_0000 + 24 = 0x2000_0018
  • 拍4：0x2000_0000 + 32 = 0x2000_0020

【例题2.3：4KB边界拆分】

Master发起：

  • 起始地址：0x0000_0FF8
  • SIZE：3
  • LENGTH：4

问：这笔事务是否合法？如果不合法，Interconnect应如何拆分？

解析：

  • 总字节 = 8 × 5 = 40字节
  • 起始地址0x0FF8，4KB边界在0x1000
  • 从0x0FF8到0x1000共8字节（0x0FF8~0x0FFF），正好1拍
  • 剩余4拍（32字节）从0x1000开始
  • 不合法，必须拆分：
    • 子事务1：地址0x0FF8，SIZE=3，LENGTH=0（1拍，8字节）
    • 子事务2：地址0x1000，SIZE=3，LENGTH=3（4拍，32字节）

【例题2.4：WRAP边界计算（完整版）】

CPU发起：

  • 起始地址：0x0000_0028
  • SIZE：3
  • LENGTH：3
  • BURST：WRAP

问：计算WRAP边界，并写出4拍地址序列。

解析：

  • 突发总大小 = 8 × 4 = 32字节
  • WRAP边界 = 32字节对齐
  • 下边界 = floor(0x28 / 32) × 32 = 0x20
  • 上边界 = 0x20 + 32 = 0x40
  • 地址序列：
    • 拍0：0x28
    • 拍1：0x28 + 8 = 0x30
    • 拍2：0x30 + 8 = 0x38
    • 拍3：0x38 + 8 = 0x40（到达上边界）→ 回绕到 0x20

完整序列：0x28, 0x30, 0x38, 0x20

【例题2.5：WRAP边界计算（非对齐起始地址）】

CPU发起：

  • 起始地址：0x0000_001A
  • SIZE：2
  • LENGTH：3
  • BURST：WRAP

问：计算边界并写出地址序列。

解析：

  • 突发总大小 = 4 × 4 = 16字节
  • 下边界 = floor(0x1A / 16) × 16 = 0x10
  • 上边界 = 0x10 + 16 = 0x20
  • 地址序列：
    • 拍0：0x1A
    • 拍1：0x1A + 4 = 0x1E
    • 拍2：0x1E + 4 = 0x22（超过上边界0x20）→ 回绕到 0x10 + (0x22 - 0x20) = 0x12
    • 拍3：0x12 + 4 = 0x16

完整序列：0x1A, 0x1E, 0x12, 0x16

⚠️ 注意：WRAP回绕时，超出的偏移量要加上下边界，不是简单回到下边界！

第二部分：数据通道与字节选通（WSTRB）

第3章 车道映射与WSTRB计算

3.1 核心理论

车道（Lane）与位宽的关系：

总线位宽 车道数 地址低位决定偏移
32-bit 4 ADDR[1:0]
64-bit 8 ADDR[2:0]
128-bit 16 ADDR[3:0]
256-bit 32 ADDR[4:0]

WSTRB计算规则：

  1. 起始车道 = 地址低位（如128-bit总线，ADDR[3:0]）
  2. 有效车道数 = 2^SIZE
  3. WSTRB中对应的bit置1
  4. 关键：WSTRB的bit0对应数据总线的byte 0（最低字节）

小端序数据分配：

  • 数据的最低字节（Byte 0）放在最低地址
  • 例如数据0xDEADBEEF，Byte 0 = 0xEF, Byte 1 = 0xBE, Byte 2 = 0xAD, Byte 3 = 0xDE

3.2 完整例题

【例题3.1：32-bit总线WSTRB计算】

Master（32-bit）写数据：

  • 起始地址：0x1000_0003
  • SIZE：1（2字节）

问：传输2字节，占用哪些车道？WSTRB是多少（4位二进制）？

解析：

  • 32-bit总线有4个车道（Lane 0~3）

  • 地址低2位 = 0x3 = 3（二进制11）

  • 从Lane 3开始，连续2个车道：Lane 3, Lane 4？❌ Lane最大为3，不能越界！

  • 这里涉及非对齐+边界处理：从地址0x3开始的2字节，实际占用Lane 3和Lane 0（跨4字节边界回绕）

  • 不对，实际上AXI的地址递增是按字节地址，不是按车道回绕：

    • 地址0x3对应的字节是总线的byte 3（Lane 3）
    • 地址0x4对应的字节是总线的byte 4，但32-bit总线只有byte 0~3
    • 所以这笔传输跨越了32-bit总线的自然边界，需要Interconnect拆分
    • 如果不拆分，实际只能传输1字节（从0x3到0x3），或者WSTRB只设置Lane 3

  • 正确理解：

    • 如果地址0x1000_0003，SIZE=1（2字节），实际传输地址0x3和0x4
    • 0x3在Lane 3，0x4已经越界到下一个32-bit字（地址0x1000_0004）
    • 这笔事务必须拆成两拍：
      • 拍0：地址0x1000_0003，传输1字节，WSTRB=1000（bit3=1）
      • 拍1：地址0x1000_0004，传输1字节，WSTRB=0001（bit0=1）

⚠️ 重要结论：AXI允许非对齐，但不能跨越自然边界（即地址+字节数不能超过2^SIZE对齐的边界）

【例题3.2：64-bit总线WSTRB计算（基础）】

GMAC（64-bit）写数据：

  • 起始地址：0x0000_000A
  • SIZE：2（4字节）

问：占用哪些车道？WSTRB是多少（8位二进制）？

解析：

  • 64-bit有8个车道（Lane 0~7）
  • 地址低3位 = 0xA & 0x7 = 0x2 = 2（二进制010）
  • 从Lane 2开始，连续4个车道：Lane 2, 3, 4, 5
  • WSTRB（8位，bit0对应Lane 0）：
    • Lane 0~1：0
    • Lane 2~5：1
    • Lane 6~7：0
    • 二进制（bit7到bit0）：00111100 = 0x3C

【例题3.3：128-bit总线WSTRB计算（非对齐）】

CPU（128-bit）写DDR：

  • 起始地址：0x8000_0005
  • SIZE：2（4字节）

问：占用哪些车道？WSTRB是多少（16位十六进制）？

解析：

  • 128-bit有16个车道（Lane 0~15）
  • 地址低4位 = 0x5
  • 从Lane 5开始，连续4个车道：Lane 5, 6, 7, 8
  • WSTRB（16位，bit0→Lane 0，bit15→Lane 15）：
    • bit5~bit8 = 1
    • 二进制：0000 0001 1110 0000
    • 十六进制：0x01E0

【例题3.4：多拍突发WSTRB序列】

CPU（128-bit）写DDR：

  • 起始地址：0x8000_0005
  • SIZE：2
  • LENGTH：1

问：写出每拍的地址和WSTRB。

解析：

  • 拍0：地址0x8000_0005，WSTRB=0x01E0（Lane 5~8）
  • 拍1：地址0x8000_0005 + 4 = 0x8000_0009
    • 地址低4位 = 0x9
    • 从Lane 9开始，连续4个车道：Lane 9, 10, 11, 12
    • WSTRB = 0xF000（bit9~bit12=1）

完整答案：

拍序 地址 WSTRB
拍0 0x8000_0005 0x01E0
拍1 0x8000_0009 0xF000

【例题3.5：WSTRB结合数据分配（小端序）】

接上题，CPU要写的数据是 0xDEADBEEF_12345678（128-bit），小端序。
问：第1拍和第2拍分别写了数据的哪些字节？

解析：

  • 原始数据按小端序分解为字节（从低到高）：
    字节序号 0 1 2 3 4 5 6 7 8~15
    值 0x78 0x56 0x34 0x12 0xEF 0xBE 0xAD 0xDE 未定义
  • 拍0（Lane 5~8）：写入Byte 5~8
    • Byte 5 = 0xBE, Byte 6 = 0xAD, Byte 7 = 0xDE, Byte 8 = 未定义（0x??）
  • 拍1（Lane 9~12）：写入Byte 9~12（均为未定义）

有效字节总数：2拍 × 4字节/拍 = 8字节有效数据

第三部分：系统拓扑与协议转换

第4章 Interconnect的角色与转换逻辑

4.1 核心理论

Interconnect的三大职责：

职责 说明 验证要点
地址译码 根据地址范围路由到对应Slave 检查地址是否准确命中目标
数据宽度转换 Master和Slave位宽不同时自动适配 检查WSTRB在转换中的正确性
协议转换 AXI→AHB→APB等跨协议转换 检查响应映射是否正确

AXI转APB的特殊性：

  • APB不支持burst（无LEN字段）
  • APB无独立的写响应通道（与传输在同一周期）
  • APB地址只能以字节为单位，无SIZE概念（实际由PSTRB控制）

4.2 完整例题

【例题4.1：数据宽度拆分（128-bit→32-bit）】

系统拓扑：

  • DMA：128-bit Master，发起写事务
  • SMC：32-bit Slave，地址范围0x1000_0000~0x1000_0FFF

DMA事务参数：

  • 起始地址：0x1000_0000
  • SIZE：4
  • LENGTH：3

问：Interconnect需要拆成多少笔子事务？每笔的SIZE和LENGTH如何设定？

解析：

  • 总字节 = 2^4 × (3+1) = 16 × 4 = 64字节
  • SMC最大SIZE = 2（2^2=4字节）
  • 每笔子事务最大能力：SIZE=2，LENGTH=0（单拍4字节）
  • 总子事务数 = 64 / 4 = 16笔
  • 每笔参数：SIZE=2, LENGTH=0
  • 地址序列：0x1000_0000, 0x0004, 0x0008, …, 0x0000_003C

【例题4.2：AXI转APB（突发拆单拍）】

系统拓扑：

  • CPU：128-bit AXI4 Master
  • APB Bridge：32-bit APB3 Slave，地址范围0x2000_0000~0x2000_00FF

CPU事务参数：

  • 起始地址：0x2000_0010
  • SIZE：2
  • LENGTH：3

问：APB不支持burst，需要拆成多少次单拍传输？写出每次的地址。

解析：

  • SIZE=2 → 每拍4字节，正好是APB的单拍能力
  • LENGTH=3 → 共4拍
  • APB无LEN字段，必须拆成4次独立的APB写操作
  • 地址序列：
    • 操作0：0x2000_0010
    • 操作1：0x2000_0014
    • 操作2：0x2000_0018
    • 操作3：0x2000_001C
  • 验证要点：检查AXI的B通道响应是否映射了APB的传输完成/错误状态

【例题4.3：数据宽度拼接（128-bit→256-bit）】

系统拓扑：

  • DMA：128-bit AXI3 Master
  • DDR：256-bit AXI4 Slave，地址范围0x8000_0000~0x8FFF_FFFF

DMA事务参数：

  • 起始地址：0x8000_0100
  • SIZE：4
  • LENGTH：1

问：DDR是256-bit总线，Interconnect如何将2拍128-bit数据拼接成1拍256-bit？

解析：

  • 2拍128-bit数据 = 32字节 = 1拍256-bit
  • 起始地址0x8000_0100，低5位=0（256-bit对齐）
  • 拼接方式：
    • 拍0的数据放到256-bit总线的低128位（byte 0~15）
    • 拍1的数据放到256-bit总线的高128位（byte 16~31）
  • 合并后的WSTRB = 32位全1（如果DMA的WSTRB全有效）
  • 验证要点：检查拼接顺序是否正确（低地址对应低字节）

【例题4.4：跨协议+跨宽度混合转换】

系统拓扑：

  • CPU：128-bit AXI4 Master
  • APB Bridge：32-bit APB3 Slave

CPU事务参数：

  • 起始地址：0x2000_0002（非对齐！）
  • SIZE：3
  • LENGTH：1

问：Interconnect需要如何处理这笔事务？

解析：

  • 第一步：检查地址对齐
    • SIZE=3 → 8字节传输
    • 起始地址0x2000_0002，低3位=2，非8字节对齐
    • 需要先拆分处理非对齐部分
  • 第二步：非对齐拆分
    • 拍0：地址0x2000_0002，需要传输从byte2开始的6字节，但会跨到0x2000_0008（8字节边界）
    • 实际处理：拆成两笔子事务
      • 子事务A：地址0x2000_0002，传输6字节（到0x2000_0007），但SIZE=3必须8字节对齐，所以实际能力受限
    • 更合理的处理：将原事务按SIZE=2（4字节）拆分
  • 第三步：转换到APB
    • 原2拍8字节 → 拆成4笔4字节 → APB需要4次单拍写
    • 地址序列：0x2000_0002, 0x2000_0006, 0x2000_000A, 0x2000_000E
  • 验证要点：检查WSTRB在每一步转换中的正确性

第四部分：AXI3与AXI4版本差异

第5章 版本差异与兼容性

5.1 核心理论

特性 AXI3 AXI4 验证影响
最大LEN 15（16拍） 255（256拍） AXI3 Master连AXI4 Slave需检查超长突发
WSTRB规则 允许未使用字节任意值 未使用字节必须为0 更严格的检查
锁定访问 支持 弃用 AXI4 Master不应发出LOCKED信号
写响应 支持WSTRB为全0 要求至少1个字节有效 空写事务非法

5.2 完整例题

【例题5.1：AXI3最大长度检查】

DMA是AXI3 Master，发起事务：

  • SIZE：4
  • LENGTH：0xF

问：这笔事务是否合法？总共传输多少字节？

解析：

  • LENGTH=15 → 实际拍数 = 15+1 = 16拍
  • AXI3最大LEN=15 → 合法（正好达到上限）
  • 总字节 = 16 × 16 = 256字节

【例题5.2：AXI3超长突发处理】

接上题，如果DMA想传输512字节（32拍），应如何拆分？

解析：

  • AXI3单笔最大16拍，512字节/16字节/拍 = 32拍
  • 最少拆成2笔：
    • 子事务1：SIZE=4, LENGTH=0xF（16拍，256字节）
    • 子事务2：SIZE=4, LENGTH=0xF（16拍，256字节）
  • 地址：
    • 子事务1：起始0x8000_0100
    • 子事务2：起始0x8000_0200
  • 验证要点：检查Interconnect是否自动拆分，且拆分后的地址连续

【例题5.3：AXI4 WSTRB严格规则】

AXI4 Slave收到一笔写事务：

  • SIZE：2（4字节）
  • WSTRB = 0xF0（8-bit总线，bit4~7有效）

问：这笔事务是否合法？为什么？

解析：

  • SIZE=2 → 4字节传输，应该有4个bit为1

  • WSTRB=0xF0 → bit4~7为1，共4个bit

  • 看起来数量对，但位置不对！

  • 对于4字节传输，有效bit应该是连续的4个bit，且与地址低位对齐

  • 如果地址低3位=0，WSTRB应为0x0F（bit0~3）

  • 如果WSTRB=0xF0（bit4~7），说明传输的4字节从byte4开始

  • 这要求地址低3位=4，且总线至少8字节宽

  • 需要具体分析：如果地址低3位=4，则合法；否则非法

  • 更典型的AXI4检查：如果SIZE=2且地址低3位=0，WSTRB必须是0x0F，不能是0x3C或0xF0

【例题5.4：AXI3→AXI4桥接兼容性】

系统中有AXI3 Master连接到AXI4 Interconnect，Master发出：

  • 起始地址：0x0000_0000
  • SIZE：3
  • LENGTH：0x1F

问：Interconnect应如何处理？

解析：

  • AXI3不支持LEN=31（超过15）
  • 如果Master发出非法长度，有两种处理方式：
    1. SLVERR：返回从机错误，拒绝事务
    2. 自动拆分：将32拍拆成2笔16拍
  • 在实际系统中，Interconnect通常会自动拆分
  • 拆分后的子事务：
    • 子事务1：LENGTH=0xF（16拍）
    • 子事务2：LENGTH=0xF（16拍）
  • 验证要点：检查拆分后的地址连续性和数据完整性

第五部分：高级主题与专项验证

第6章 原子操作与ID管理

6.1 核心理论

独占访问（Exclusive）：

  • 用于多核系统的信号量操作
  • Master发出Exclusive标记（AxLOCK=1）
  • Slave需要监视地址，检查在操作期间是否被其他Master修改
  • 返回EXOKAY表示成功，OKAY表示失败

ID与乱序完成：

  • 每个事务有AXID（AWID/ARID）
  • Slave可以不按顺序返回响应（RID/BID匹配）
  • 允许Master同时发出多个未完成事务（Outstanding）

6.2 完整例题

【例题6.1：独占访问验证】

CPU1发起独占写操作到地址0x1000_0000，在操作期间CPU2也写入了同一地址。
问：CPU1的独占操作应返回什么响应？

解析：

  • CPU1的Exclusive操作应返回EXOKAY（表示操作成功）还是OKAY（表示失败）？
  • 正确答案：OKAY（失败）
  • 因为地址在独占监视期间被其他Master修改，独占条件被破坏
  • 验证时需要构造这种竞争场景

【例题6.2：乱序完成与ID匹配】

Master发出两笔读事务：

  • 事务A：ARID=1，地址0x1000
  • 事务B：ARID=2，地址0x2000

Slave返回数据：

  • 先返回事务B的数据（RID=2）
  • 后返回事务A的数据（RID=1）

问：Master能否正确处理？Scoreboard应如何验证？

解析：

  • Master必须能正确处理乱序返回（AXI支持）
  • Master通过RID区分事务
  • Scoreboard的验证方法：
    1. 维护一个待处理事务队列，记录每个ARID的预期数据
    2. 收到R数据时，用RID查找对应的预期数据
    3. 比对实际数据和预期数据
    4. 检查是否所有事务都完成（没有遗漏）

【综合题A：完整系统事务分析】

系统拓扑：

  • Master A：128-bit AXI4
  • Master B：64-bit AXI3
  • Slave X：32-bit AXI3（地址0x0000_0000~0x0000_FFFF）
  • Slave Y：256-bit AXI4（地址0x8000_0000~0x8FFF_FFFF）
  • APB Bridge：32-bit APB3（地址0x2000_0000~0x2000_00FF）

事务1：Master A写Slave X

  • 地址：0x0000_000A
  • SIZE：3
  • LENGTH：2
  • BURST：INCR
  • 数据：0xAABBCCDD_EEFF0011

问：

  1. 每拍地址是多少？
  2. 每拍的WSTRB是多少（128-bit总线）？
  3. 需要拆成多少笔子事务给Slave X（32-bit）？
  4. 拆分后每笔的地址和WSTRB？

解析：

  • 1. 每拍地址：

    • 拍0：0x0000_000A
    • 拍1：0x0000_000A + 8 = 0x0000_0012
    • 拍2：0x0000_0012 + 8 = 0x0000_001A

  • 2. 每拍WSTRB（128-bit=16车道）：

    • 拍0（0x0A，低4位=10）：
      • SIZE=3（8字节），从Lane 10开始连续8个车道：Lane 10~17
      • 但16车道最大Lane 15，Lane 17不存在！
      • 所以拍0跨越了128-bit边界（0x0A到0x11），需要拆分
      • 实际处理：拆成两拍 → 0x0A传6字节（Lane 10~15），0x10传2字节（Lane 0~1）
    • 拍1（0x12，低4位=2）：
      • 从Lane 2开始连续8个车道：Lane 2~9
      • WSTRB = 0x03FC（二进制 0000 0011 1111 1100）
    • 拍2（0x1A，低4位=10）：
      • 同拍0，需要拆分

  • 3. 拆给Slave X（32-bit）：

    • 原事务24字节，Slave X每拍4字节
    • 需要拆分6笔单拍写（如果考虑非对齐会更复杂）
