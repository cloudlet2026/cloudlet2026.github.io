第一部分：AXI 协议基础架构

第1章 五通道与握手机制

1.1 核心理论

五个独立通道：

  • 读地址通道（AR）：发送读请求地址和控制信号
  • 读数据通道（R）：返回读数据和响应
  • 写地址通道（AW）：发送写请求地址和控制信号
  • 写数据通道（W）：发送写数据
  • 写响应通道（B）：返回写响应

通道分离的目的是流水线化——地址可以在前一笔数据未完成时提前发出。

VALID/READY 双向握手：

规则 说明
VALID由发送方控制 一旦拉高，必须保持到握手成功
READY由接收方控制 可随时拉高或拉低
握手成功条件 VALID && READY 同时为高
黄金法则 VALID不能依赖于READY（否则死锁）

1.2 完整例题

【例题1.1：握手时序分析】

某AXI总线上，VALID在T1周期上升沿拉高，READY在T3周期上升沿拉高。请问数据在哪个周期完成传输？如果READY一直不拉高，VALID应该保持多久？

解析：

  • 握手发生在T3周期（VALID && READY同时为高）
  • 如果READY一直不拉高，VALID必须无限期保持，直到握手完成
  • 验证时，需要构造READY延迟多个周期的场景，检查DUT是否能正确处理

【例题1.2：握手依赖风险识别】

下面这段RTL逻辑是否存在死锁风险？为什么？

解析：

  • 存在协议风险，也可能形成死锁。这段逻辑让VALID组合依赖READY：只有READY为高时才愿意发起VALID
  • 如果接收端的READY策略也是“看到VALID后才拉高READY”，两端互相等待，握手永远不会发生
  • 即使没有死锁，这种组合依赖也容易形成跨模块组合路径，破坏时序收敛
  • 正确做法：VALID应由发送方根据“有数据/有请求”的状态产生；一旦VALID拉高，在VALID && READY握手前保持稳定
  • READY可以根据接收方缓冲区、仲裁和背压状态独立拉高或拉低

第2章 突发传输（Burst）核心计算

2.1 核心理论

三个核心参数：

参数 含义 计算公式
SIZE（AxSIZE） 每拍传输的最大字节数 字节数 = 2^SIZE
LEN（AxLEN） 突发长度编码 实际拍数 = LEN + 1
总字节数 一次突发的最大数据量 最大字节数 = 2^SIZE × (LEN + 1)

下文例题中保留“LENGTH”这个口语写法，它对应协议信号里的 AxLEN 编码，而不是“实际拍数”本身。

三种BURST类型：

类型 地址计算 典型用途
INCR 每拍地址 += 2^SIZE 普通内存访问
WRAP 递增到上边界后回绕 Cache line填充
FIXED 所有拍地址相同 FIFO访问

4KB边界规则：

  • 任何突发不能跨越4KB（0x1000）地址边界
  • 合规的Master应保证自己发出的突发不跨4KB
  • Interconnect或Bridge如果需要向下游转发，必须避免向下游发出跨4KB的burst；具体实现可以拆分、报错或拒绝非法事务，不能简单假设所有互连都会“自动修好”

2.2 完整例题

【例题2.1：基础突发计算】

Master发起一笔写事务：

  • 起始地址：0x2000_0000
  • SIZE：3
  • LENGTH：4

问：每拍传输多少字节？共几拍？总字节多少？

解析：

  • 每拍字节 = 2^3 = 8字节
  • 总拍数 = 4 + 1 = 5拍
  • 总字节 = 8 × 5 = 40字节

【例题2.2：地址序列计算（INCR）】

接上题，写出每拍的地址。

解析：

  • 拍0：0x2000_0000
  • 拍1：0x2000_0000 + 8 = 0x2000_0008
  • 拍2：0x2000_0000 + 16 = 0x2000_0010
  • 拍3：0x2000_0000 + 24 = 0x2000_0018
  • 拍4：0x2000_0000 + 32 = 0x2000_0020

【例题2.3：4KB边界拆分】

Master发起：

  • 起始地址：0x0000_0FF8
  • SIZE：3
  • LENGTH：4

问：这笔事务是否合法？如果不合法，Interconnect应如何拆分？

解析：

  • 总字节 = 8 × 5 = 40字节
  • 起始地址0x0FF8，4KB边界在0x1000
  • 从0x0FF8到0x1000共8字节（0x0FF8~0x0FFF），正好1拍
  • 剩余4拍（32字节）从0x1000开始
  • 不合法，必须拆分：
    • 子事务1：地址0x0FF8，SIZE=3，LENGTH=0（1拍，8字节）
    • 子事务2：地址0x1000，SIZE=3，LENGTH=3（4拍，32字节）

【例题2.4：WRAP边界计算（完整版）】

CPU发起：

  • 起始地址：0x0000_0028
  • SIZE：3
  • LENGTH：3
  • BURST：WRAP

问：计算WRAP边界，并写出4拍地址序列。

解析：

  • 突发总大小 = 8 × 4 = 32字节
  • WRAP边界 = 32字节对齐
  • 下边界 = floor(0x28 / 32) × 32 = 0x20
  • 上边界 = 0x20 + 32 = 0x40
  • 地址序列：
    • 拍0：0x28
    • 拍1：0x28 + 8 = 0x30
    • 拍2：0x30 + 8 = 0x38
    • 拍3：0x38 + 8 = 0x40（到达上边界）→ 回绕到 0x20

完整序列：0x28, 0x30, 0x38, 0x20

【例题2.5：WRAP边界计算（非对齐起始地址）】

CPU发起：

  • 起始地址：0x0000_001A
  • SIZE：2
  • LENGTH：3
  • BURST：WRAP

问：这笔WRAP事务是否合法？如果只按数学回绕计算，地址序列是什么？

解析：

  • 突发总大小 = 4 × 4 = 16字节
  • 下边界 = floor(0x1A / 16) × 16 = 0x10
  • 上边界 = 0x10 + 16 = 0x20
  • 但WRAP burst的起始地址必须按每拍传输字节数对齐；SIZE=2表示4字节，0x1A不是4字节对齐地址
  • 因此这笔协议上非法，验证环境应检查Master不会发出这种WRAP事务
  • 如果只是练习回绕公式，可以得到数学序列：
  • 地址序列：
    • 拍0：0x1A
    • 拍1：0x1A + 4 = 0x1E
    • 拍2：0x1E + 4 = 0x22（超过上边界0x20）→ 回绕到 0x10 + (0x22 - 0x20) = 0x12
    • 拍3：0x12 + 4 = 0x16

完整序列：0x1A, 0x1E, 0x12, 0x16

⚠️ 注意：WRAP回绕时，超出的偏移量要加上下边界，不是简单回到下边界！

第二部分：数据通道与字节选通（WSTRB）

第3章 车道映射与WSTRB计算

3.1 核心理论

车道（Lane）与位宽的关系：

总线位宽 车道数 地址低位决定偏移
32-bit 4 ADDR[1:0]
64-bit 8 ADDR[2:0]
128-bit 16 ADDR[3:0]
256-bit 32 ADDR[4:0]

WSTRB计算规则：

  1. 数据总线的每8bit对应1个byte lane，WSTRB[n]控制WDATA[(8n+7):8n]
  2. 起始车道通常由地址低位决定：例如128-bit总线有16个byte lane，使用ADDR[3:0]定位起始lane
  3. 对齐且完整的窄传输中，有效车道数通常为2^SIZE
  4. 对非对齐写传输，首拍可能只使能从起始地址到本数据拍边界之间的部分byte lane；不要简单地让WSTRB在同一拍内“回绕”
  5. 关键：WSTRB的bit0对应数据总线的byte 0（最低字节），低地址数据放在低编号byte lane

小端序数据分配：

  • 数据的最低字节（Byte 0）放在最低地址
  • 例如数据0xDEADBEEF，Byte 0 = 0xEF, Byte 1 = 0xBE, Byte 2 = 0xAD, Byte 3 = 0xDE

3.2 完整例题

【例题3.1：32-bit总线WSTRB计算】

Master（32-bit）写数据：

  • 起始地址：0x1000_0003
  • SIZE：1（2字节）

问：传输2字节，占用哪些车道？WSTRB是多少（4位二进制）？

解析：

  • 32-bit总线有4个车道（Lane 0~3）
  • 地址低2位 = 0x3 = 3（二进制11）
  • 从Lane 3开始，SIZE=1期望最多覆盖2字节，但同一拍不能把WSTRB从Lane 3回绕到Lane 0
  • 因此首拍只能使能Lane 3，WSTRB = 4’b1000
  • 后续地址推进由burst地址规则决定：非对齐INCR burst通常只有首拍非对齐，后续拍按transfer size对齐推进
  • 验证时应检查：
    • WSTRB不能在同一数据拍内跨数据总线边界回绕
    • WSTRB置1的位置必须落在该拍地址允许访问的byte lane范围内
    • 不能用“$countones(WSTRB)一定等于2^SIZE”作为唯一规则，因为非对齐首拍和部分写可能使有效字节数少于2^SIZE

⚠️ 重要结论：AXI允许非对齐INCR传输，但WSTRB不能在一个数据拍内跨总线lane边界回绕；4KB边界规则仍然必须满足。

【例题3.2：64-bit总线WSTRB计算（基础）】

GMAC（64-bit）写数据：

  • 起始地址：0x0000_000A
  • SIZE：2（4字节）

问：占用哪些车道？WSTRB是多少（8位二进制）？

解析：

  • 64-bit有8个车道（Lane 0~7）
  • 地址低3位 = 0xA & 0x7 = 0x2 = 2（二进制010）
  • 从Lane 2开始，连续4个车道：Lane 2, 3, 4, 5
  • WSTRB（8位，bit0对应Lane 0）：
    • Lane 0~1：0
    • Lane 2~5：1
    • Lane 6~7：0
    • 二进制（bit7到bit0）：00111100 = 0x3C

【例题3.3：128-bit总线WSTRB计算（非对齐）】

CPU（128-bit）写DDR：

  • 起始地址：0x8000_0005
  • SIZE：2（4字节）

问：占用哪些车道？WSTRB是多少（16位十六进制）？

解析：

  • 128-bit有16个车道（Lane 0~15）
  • 地址低4位 = 0x5
  • 从Lane 5开始，连续4个车道：Lane 5, 6, 7, 8
  • WSTRB（16位，bit0→Lane 0，bit15→Lane 15）：
    • bit5~bit8 = 1
    • 二进制：0000 0001 1110 0000
    • 十六进制：0x01E0

【例题3.4：多拍突发WSTRB序列】

CPU（128-bit）写DDR：

  • 起始地址：0x8000_0005
  • SIZE：2
  • LENGTH：1

问：写出每拍的地址和WSTRB。

解析：

  • 拍0：地址0x8000_0005，WSTRB=0x01E0（Lane 5~8）
  • 拍1：地址0x8000_0005 + 4 = 0x8000_0009
    • 地址低4位 = 0x9
    • 从Lane 9开始，连续4个车道：Lane 9, 10, 11, 12
    • WSTRB = 0x1E00（bit9~bit12=1）

完整答案：

拍序 地址 WSTRB
拍0 0x8000_0005 0x01E0
拍1 0x8000_0009 0x1E00

【例题3.5：WSTRB结合数据分配（小端序）】

接上题，CPU要写入连续8字节数据 0xDEADBEEF_12345678，小端序。
问：第1拍和第2拍分别把哪些数据字节放到哪些lane？

解析：

  • 原始数据按小端序分解为字节（从低地址到高地址）：
    数据字节序号 0 1 2 3 4 5 6 7
    值 0x78 0x56 0x34 0x12 0xEF 0xBE 0xAD 0xDE
  • 拍0（地址0x8000_0005，Lane 5~8）：
    • Lane 5 = 0x78，Lane 6 = 0x56，Lane 7 = 0x34，Lane 8 = 0x12
  • 拍1（地址0x8000_0009，Lane 9~12）：
    • Lane 9 = 0xEF，Lane 10 = 0xBE，Lane 11 = 0xAD，Lane 12 = 0xDE

有效字节总数：2拍 × 4字节/拍 = 8字节有效数据

第三部分：系统拓扑与协议转换

第4章 Interconnect的角色与转换逻辑

4.1 核心理论

Interconnect的三大职责：

职责 说明 验证要点
地址译码 根据地址范围路由到对应Slave 检查地址是否准确命中目标
数据宽度转换 Master和Slave位宽不同时自动适配 检查WSTRB在转换中的正确性
协议转换 AXI→AHB→APB等跨协议转换 检查响应映射是否正确

AXI转APB的特殊性：

  • APB不支持burst（无LEN字段）
  • APB无AXI式独立B通道；错误状态通过PSLVERR随APB传输返回，再由Bridge映射到AXI响应
  • APB无AXI式SIZE概念；APB4可用PSTRB表达字节选通，APB3通常需要Bridge用读改写或更小粒度访问保护未选中字节

4.2 完整例题

【例题4.1：数据宽度拆分（128-bit→32-bit）】

系统拓扑：

  • DMA：128-bit Master，发起写事务
  • SMC：32-bit Slave，地址范围0x1000_0000~0x1000_0FFF

DMA事务参数：

  • 起始地址：0x1000_0000
  • SIZE：4
  • LENGTH：3

问：Interconnect需要拆成多少笔子事务？每笔的SIZE和LENGTH如何设定？

解析：

  • 总字节 = 2^4 × (3+1) = 16 × 4 = 64字节
  • SMC最大SIZE = 2（2^2=4字节）
  • 每笔子事务最大能力：SIZE=2，LENGTH=0（单拍4字节）
  • 总子事务数 = 64 / 4 = 16笔
  • 每笔参数：SIZE=2, LENGTH=0
  • 地址序列：0x1000_0000, 0x1000_0004, 0x1000_0008, …, 0x1000_003C

【例题4.2：AXI转APB（突发拆单拍）】

系统拓扑：

  • CPU：128-bit AXI4 Master
  • APB Bridge：32-bit APB3 Slave，地址范围0x2000_0000~0x2000_00FF

CPU事务参数：

  • 起始地址：0x2000_0010
  • SIZE：2
  • LENGTH：3

问：APB不支持burst，需要拆成多少次单拍传输？写出每次的地址。

解析：

  • SIZE=2 → 每拍4字节，正好是APB的单拍能力
  • LENGTH=3 → 共4拍
  • APB无LEN字段，必须拆成4次独立的APB写操作
  • 地址序列：
    • 操作0：0x2000_0010
    • 操作1：0x2000_0014
    • 操作2：0x2000_0018
    • 操作3：0x2000_001C
  • 验证要点：检查AXI的B通道响应是否映射了APB的传输完成/错误状态

【例题4.3：数据宽度拼接（128-bit→256-bit）】

系统拓扑：

  • DMA：128-bit AXI3 Master
  • DDR：256-bit AXI4 Slave，地址范围0x8000_0000~0x8FFF_FFFF

DMA事务参数：

  • 起始地址：0x8000_0100
  • SIZE：4
  • LENGTH：1

问：DDR是256-bit总线，Interconnect如何将2拍128-bit数据拼接成1拍256-bit？

解析：

  • 2拍128-bit数据 = 32字节 = 1拍256-bit
  • 起始地址0x8000_0100，低5位=0（256-bit对齐）
  • 拼接方式：
    • 拍0的数据放到256-bit总线的低128位（byte 0~15）
    • 拍1的数据放到256-bit总线的高128位（byte 16~31）
  • 合并后的WSTRB = 32位全1（如果DMA的WSTRB全有效）
  • 验证要点：检查拼接顺序是否正确（低地址对应低字节）

【例题4.4：跨协议+跨宽度混合转换】

系统拓扑：

  • CPU：128-bit AXI4 Master
  • APB Bridge：32-bit APB3 Slave

CPU事务参数：

  • 起始地址：0x2000_0002（非对齐！）
  • SIZE：3
  • LENGTH：1

问：Interconnect需要如何处理这笔事务？

解析：

  • 第一步：识别AXI侧事务
    • SIZE=3 → 每拍最多8字节
    • LENGTH=1 → 2拍
    • 起始地址0x2000_0002非8字节对齐，因此首拍是非对齐写
  • 第二步：生成AXI侧可观察行为
    • 首拍从地址0x2000_0002开始，128-bit总线下可使用Lane 2~9，WSTRB=0x03FC
    • Aligned_Address = floor(0x2000_0002 / 8) × 8 = 0x2000_0000
    • 第二拍地址 = 0x2000_0000 + 8 = 0x2000_0008，可使用Lane 8~15，WSTRB=0xFF00
  • 第三步：转换到APB
    • APB3本身没有PSTRB，常见APB4才有PSTRB；如果下游是APB3，Bridge通常需要读改写或拆成更小粒度访问来保护未选中字节
    • 如果Bridge面向32-bit APB4并支持PSTRB，可拆成若干32-bit单拍访问，例如覆盖0x2000_0000、0x2000_0004、0x2000_0008、0x2000_000C等对齐字地址，并按PSTRB屏蔽无效字节
  • 验证要点：检查WSTRB在每一步转换中的正确性

第四部分：AXI3与AXI4版本差异

第5章 版本差异与兼容性

5.1 核心理论

特性 AXI3 AXI4 验证影响
最大LEN 15（最多16拍） INCR最多255（256拍）；WRAP仍只能是2/4/8/16拍 AXI4 Master接AXI3 Slave时，Bridge需要限制或拆分超长INCR burst
写数据ID 有WID，支持写数据交织 取消WID，不支持写数据交织 AXI3到AXI4桥接要去交织并保证写数据顺序
锁定访问 支持locked/exclusive等编码 简化为exclusive，locked访问被移除 AXI4 Master不应发出AXI3式LOCKED事务
QoS/Region 不支持或不作为标准信号 增加AxQOS、AxREGION等系统级提示 验证时要检查默认值、透传或丢弃策略
WSTRB检查 都有WSTRB 都有WSTRB 不应把“未使用字节任意值/必须为0”作为AXI3/AXI4核心差异；关键是WSTRB与地址、SIZE、总线宽度、部分写策略一致

5.2 完整例题

【例题5.1：AXI3最大长度检查】

DMA是AXI3 Master，发起事务：

  • SIZE：4
  • LENGTH：0xF

问：这笔事务是否合法？总共传输多少字节？

解析：

  • LENGTH=15 → 实际拍数 = 15+1 = 16拍
  • AXI3最大LEN=15 → 合法（正好达到上限）
  • 总字节 = 16 × 16 = 256字节

【例题5.2：AXI3超长突发处理】

接上题，如果DMA想传输512字节（32拍），应如何拆分？

解析：

  • AXI3单笔最大16拍，512字节/16字节/拍 = 32拍
  • 最少拆成2笔：
    • 子事务1：SIZE=4, LENGTH=0xF（16拍，256字节）
    • 子事务2：SIZE=4, LENGTH=0xF（16拍，256字节）
  • 地址：
    • 子事务1：起始0x8000_0100
    • 子事务2：起始0x8000_0200
  • 验证要点：检查发起端、DMA描述符生成逻辑或协议Bridge是否按AXI3限制拆分，且拆分后的地址连续

【例题5.3：WSTRB与地址对齐检查】

AXI Slave收到一笔写事务：

  • SIZE：2（4字节）
  • WSTRB = 0xF0（8-bit总线，bit4~7有效）

问：这笔事务是否合法？为什么？

解析：

  • SIZE=2 → 4字节传输，应该有4个bit为1

  • WSTRB=0xF0 → bit4~7为1，共4个bit

  • 看起来数量对，但位置不对！

  • 对于4字节传输，有效bit应该是连续的4个bit，且与地址低位对齐

  • 如果地址低3位=0，WSTRB应为0x0F（bit0~3）

  • 如果WSTRB=0xF0（bit4~7），说明传输的4字节从byte4开始

  • 这要求地址低3位=4，且总线至少8字节宽

  • 需要具体分析：如果地址低3位=4，则合法；否则非法

  • 更典型的检查：如果SIZE=2且地址低3位=0，WSTRB应落在bit0~bit3范围内；如果地址低3位=4，WSTRB=0xF0才匹配

【例题5.4：非法LEN与桥接场景区分】

系统中有AXI3 Master连接到AXI4 Interconnect，Master发出：

  • 起始地址：0x0000_0000
  • SIZE：3
  • LENGTH：0x1F

问：Interconnect应如何处理？

解析：

  • AXI3不支持LEN=31（超过15）
  • 如果这确实是一个AXI3 Master直接发出的事务，Master本身已经违反协议，Slave/Interconnect通常只能报错或拒绝，不能把非法输入当成合规场景
  • 如果真实场景是“AXI4 Master通过Bridge访问AXI3 Slave”，Bridge才应该在AXI3侧拆成合法子事务：
    • 子事务1：LENGTH=0xF（16拍）
    • 子事务2：LENGTH=0xF（16拍）
  • 验证要点：区分“上游非法事务检查”和“合法协议转换拆分”，不要把AXI3 Master发出LEN=0x1F当成正常激励

第6章 原子操作与ID管理

6.1 核心理论

独占访问（Exclusive）：

  • 用于多核系统的信号量操作
  • Master发出Exclusive标记（AxLOCK=1）
  • Slave需要监视地址，检查在操作期间是否被其他Master修改
  • 返回EXOKAY表示成功，OKAY表示失败

ID与乱序完成：

  • 每个事务有AXID（AWID/ARID）
  • 不同ID之间可以乱序返回；同一ID内必须保持协议要求的顺序
  • 读数据通过RID匹配事务，写响应通过BID匹配写事务
  • 允许Master同时发出多个未完成事务（Outstanding）

6.2 完整例题

【例题6.1：独占访问验证】

CPU1发起独占写操作到地址0x1000_0000，在操作期间CPU2也写入了同一地址。
问：CPU1的独占操作应返回什么响应？

解析：

  • CPU1的Exclusive操作应返回EXOKAY（表示操作成功）还是OKAY（表示失败）？
  • 正确答案：OKAY（失败）
  • 因为地址在独占监视期间被其他Master修改，独占条件被破坏
  • 验证时需要构造这种竞争场景

【例题6.2：乱序完成与ID匹配】

Master发出两笔读事务：

  • 事务A：ARID=1，地址0x1000
  • 事务B：ARID=2，地址0x2000

Slave返回数据：

  • 先返回事务B的数据（RID=2）
  • 后返回事务A的数据（RID=1）

问：Master能否正确处理？Scoreboard应如何验证？

解析：

  • Master必须能正确处理不同ID之间的乱序返回（AXI支持）
  • Master通过RID区分读事务；如果两笔读事务使用同一个ARID，则R通道返回顺序不能随意颠倒
  • Scoreboard的验证方法：
    1. 维护一个待处理事务队列，记录每个ARID的预期数据
    2. 收到R数据时，用RID查找对应的预期数据
    3. 比对实际数据和预期数据
    4. 检查是否所有事务都完成（没有遗漏）

【综合题A：完整系统事务分析】

系统拓扑：

  • Master A：128-bit AXI4
  • Master B：64-bit AXI3
  • Slave X：32-bit AXI3（地址0x0000_0000~0x0000_FFFF）
  • Slave Y：256-bit AXI4（地址0x8000_0000~0x8FFF_FFFF）
  • APB Bridge：32-bit APB3（地址0x2000_0000~0x2000_00FF）

事务1：Master A写Slave X

  • 地址：0x0000_000A
  • SIZE：3
  • LENGTH：2
  • BURST：INCR
  • 数据：0xAABBCCDD_EEFF0011

问：

  1. 每拍地址是多少？
  2. 每拍的WSTRB是多少（128-bit总线）？
  3. 需要拆成多少笔子事务给Slave X（32-bit）？
  4. 拆分后每笔的地址和WSTRB？

解析：

  • 1. 每拍地址：

    • 拍0：0x0000_000A
    • Number_Bytes = 2^3 = 8，Aligned_Address = floor(0x0000_000A / 8) × 8 = 0x0000_0008
    • INCR burst只有首拍可以从非对齐地址开始，后续拍按传输大小对齐推进
    • 拍1：0x0000_0008 + 8 = 0x0000_0010
    • 拍2：0x0000_0008 + 16 = 0x0000_0018

  • 2. 每拍WSTRB（128-bit=16车道）：

    • 拍0（0x0A，低4位=10）：
      • 首拍从Lane 10开始，到当前128-bit数据拍末尾Lane 15结束
      • WSTRB = 0xFC00（二进制 1111 1100 0000 0000）
    • 拍1（0x10，低4位=0）：
      • 从Lane 0开始连续8个车道：Lane 0~7
      • WSTRB = 0x00FF（二进制 0000 0000 1111 1111）
    • 拍2（0x18，低4位=8）：
      • 从Lane 8开始连续8个车道：Lane 8~15
      • WSTRB = 0xFF00（二进制 1111 1111 0000 0000）

  • 3. 拆给Slave X（32-bit）：

    • AXI侧3拍最多承载24字节，但首拍非对齐，按上述WSTRB实际覆盖地址0x0000_000A~0x0000_001F，共22个有效字节
    • Slave X每拍4字节，32-bit对齐字地址覆盖：0x0000_0008、0x0000_000C、0x0000_0010、0x0000_0014、0x0000_0018、0x0000_001C
    • 因此需要拆成6笔32-bit单拍写，首笔WSTRB=4’b1100，其余5笔WSTRB=4’b1111

  • 4. 拆分后每笔地址和WSTRB：

    子事务 地址 32-bit WSTRB 覆盖字节
    0 0x0000_0008 4’b1100 0x0000_000A~0x0000_000B
    1 0x0000_000C 4’b1111 0x0000_000C~0x0000_000F
    2 0x0000_0010 4’b1111 0x0000_0010~0x0000_0013
    3 0x0000_0014 4’b1111 0x0000_0014~0x0000_0017
    4 0x0000_0018 4’b1111 0x0000_0018~0x0000_001B
    5 0x0000_001C 4’b1111 0x0000_001C~0x0000_001F
