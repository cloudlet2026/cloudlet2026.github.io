在航空电子系统中，机载总线网络承担着飞控、航电、导航、任务设备、维护记录等多个系统之间的数据交换。常见接口包括 AFDX/ARINC664、ARINC429、MIL-STD-1553B、CAN、RS422 和离散量 IO。随着这些接口逐渐被集成到一颗 SoC 或网络交换/记录芯片中，片上互联就从“内部连线”变成了影响系统确定性、隔离性和可验证性的关键基础设施。

本文以一个航空机载总线网络交换与记录 SoC为工程例程，讨论如何不用 UVM，仅用 CPU 裸机软件 + DMA + SystemVerilog testbench 的软硬协同方式，对 NIC400 互联做功能仿真验证。

  • 从机载总线网络应用出发介绍 NIC400 验证目标；

  • 总结不用 UVM 时仍然能保持可追溯、可复现、可调试的方法。

1. 为什么机载总线网络 SoC 更需要测 NIC400

在很多芯片项目里，NIC400 被看作可配置互联 IP，只要 ARM 工具生成、端口连上、CPU 能访问内存，大家就容易低估它的验证风险。

但在机载总线网络设备里，NIC400 上跑的不是单一软件访问，而是多类流量混合：

  • AFDX 接口持续接收以太网帧并写入 DDR ring buffer；
  • ARINC429 通道周期性写入 label 数据；
  • MIL-STD-1553B BC/RT/BM 控制器读写命令表和消息缓冲区；
  • System DMA 在 SRAM、DDR、维护日志区之间搬运数据；
  • CPU 配置路由表、过滤表、时间同步寄存器、安全监控寄存器；
  • Trace/Log DMA 在后台写飞行记录或维护记录；
  • DDR 控制器可能因为刷新、仲裁、ECC 处理产生 backpressure；
  • APB 低速寄存器访问必须保持顺序，并且不能被高带宽流量饿死。

这些流量同时存在时，NIC400 的验证目标不只是“AXI 协议没违规”，而是：

  1. 地址路由必须正确；
  2. 多 master 响应不能串线；
  3. DDR 高压力下关键 APB 访问不能超时；
  4. AFDX/1553/CAN 等业务帧不能因为互联 backpressure 被错误覆盖；
  5. 非法访问、ECC 错误、APB 错误不能污染其他正常事务；
  6. 测试结果要能形成需求到用例再到日志的证据链。

这就是机载总线网络应用中验证 NIC400 的现实意义。

20.4 不同业务看不同重点

  • AFDX：看 descriptor status、payload ring、帧顺序、DDR backpressure；
  • 1553B：看小消息 WSTRB、slot 边界、APB 配置顺序；
  • ARINC429/CAN：看 APB PREADY/PSLVERR、FIFO 状态寄存器；
  • Trace DMA：看 QoS、后台流量是否影响关键路径；
  • CPU Time/Safety：看最大延迟和授权访问；
  • DDR：看 ECC、R/B 返回、outstanding 是否归零。

20.5 Cache 和 mailbox 规则必须前置

机载总线网络 SoC 的 CPU 很可能有 cache。验证时要明确：

  • mailbox 区域必须 uncached；
  • DMA 源 buffer 启动前 flush；
  • DMA 目的 buffer 校验前 invalidate；
  • AFDX/1553 descriptor ring 建议 uncached；
  • APB 寄存器必须 device attribute；
  • DDR 帧缓存如果 cacheable，要覆盖 cache on/off 两种路径。

很多所谓 NIC400 数据错，最后其实是 cache 维护或 descriptor attribute 错。
