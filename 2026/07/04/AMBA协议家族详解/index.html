一、AMBA是什么？为什么需要它？

AMBA（Advanced Microcontroller Bus Architecture）是ARM公司制定的片上系统（SoC）互联标准。它的核心价值在于：

  • 解耦：让CPU、DMA、外设等IP模块可以独立设计，通过标准总线互联。
  • 复用：只要符合AMBA接口，任何IP都可以即插即用。
  • 性能：不同层次的协议针对不同场景优化，从低速配置到高速数据搬运全覆盖。

八、APB 状态机（官方定义）

APB3 传输由一个三状态状态机控制：

  • IDLE：默认状态，PSEL=0, PENABLE=0
  • SETUP：仅持续1个周期，PSEL=1, PENABLE=0，地址/控制信号在此阶段建立
  • ACCESS：持续1个或多个周期，PSEL=1, PENABLE=1，数据在此阶段传输；若PREADY=0则在当前状态自旋等待

九、APB四种典型时序波形图

① 无等待读传输（PREADY=1）

  • T1上升沿：进入SETUP，PSEL=1，PENABLE=0，地址ADDR被驱动
  • T2上升沿：进入ACCESS，PENABLE=1，从设备采样地址，因为PREADY=1，在T2周期内从设备将数据放到PRDATA上
  • T3上升沿：主机采样PRDATA，PSEL和PENABLE同时拉低，传输结束

② 带等待的读传输（PREADY=0插入2拍等待）

  • T1→T2：SETUP阶段（1拍）
  • T2→T3→T4：ACCESS阶段，PREADY连续为0，总线被强制扩展，PRDATA上的数据无效
  • T4→T5：PREADY在T4周期拉高，从设备在T4周期内将有效数据放到PRDATA上，T5上升沿被主机采样

③ 无等待写传输（PREADY=1）

  • 与读传输类似，区别是PWRITE=1，写数据在SETUP阶段就已经驱动到PWDATA上
  • T2上升沿进入ACCESS，从设备在T2周期内将PWDATA写入内部寄存器

④ 带PSLVERR错误的传输

  • PSLVERR只在ACCESS阶段的最后一个周期有效（即PENABLE=1且PREADY=1的周期）
  • 如果PSLVERR=1：
  • 读操作：PRDATA上的数据可能无效，主机应忽略
  • 写操作：从设备不应更新内部寄存器

十二、AHB-Lite 架构特点

AHB-Lite是AHB的单主设备版本，去掉了多主设备场景下的仲裁逻辑，适用于只有一个Master的简单SoC系统。它保留了AHB的核心特性：

  • 流水线传输：地址相位和数据相位重叠，提高总线利用率
  • 突发传输支持：支持SINGLE、INCR、WRAP等多种突发类型
  • 从设备反压：通过HREADY信号插入等待周期

十三、AHB-Lite 传输基础

AHB-Lite的每次传输分为两个相位：

相位 描述 持续周期

数据相位 读操作时Slave驱动HRDATA，写操作时Master驱动HWDATA 至少1拍（可扩展）

十六、AXI4 握手时序详解

三种握手方式

① VALID 先变高 READY 后变高

② READY 先变高 VALID 后变高

③ VALID 和 READY 信号同时变高

AXI协议规范

  • 协议的唯一铁律：VALID信号不可以依赖READY信号。也就是说，发送方（Source）只要把数据/地址准备好了，就必须把VALID拉高，不能等看到对方的READY为高再拉VALID（否则会引发死锁风险）。
  • 协议的允许范围：VALID先高、READY先高、同时高，在协议层面100%合法且等效。协议只关心"VALID和READY同时为高的那个周期，数据被成功传输"。

工程实践

这取决于当前信号是谁发出来的。

发送方角色 典型通道（读/写） 数据/地址来源 工程上常用的策略 原因
Master（主机） AW（写地址）、W（写数据）、AR（读地址） CPU/DMA内部逻辑，响应极快，一个周期内就能算出来。 倾向于"同时拉高"或"READY先高" 主机通常频率高、逻辑简单，为了追求极致带宽，希望一拍之内完成握手，绝不多等。
Slave（从机） R（读数据）、B（写响应） 外部存储器（DDR/Flash）、慢速外设、内部RAM（可能需多个周期读出）。 几乎全是"VALID先拉高" 从机不知道何时能取到数据。当它终于取到数据放在总线上时，必须第一时间把VALID置1，然后等着主机（Master）腾出手来把READY拉高把数据拿走。

场景对比

1. 什么时候用 “READY先高"或"同时高”？（发送方能力极强）

  • 场景：Master发起的通道（AW写地址、W写数据、AR读地址）。
  • 代码写法：Master在发出地址/数据的那一拍，同时把VALID拉高。Slave端通常为了不拖慢Master，会一直保持READY为高（或者尽早拉高）。
  • 效果：实现零等待（Zero Wait State）传输，一个时钟周期搞定。

2. 什么时候用 “VALID先拉高”？（发送方能力受限，需要等待）

  • 场景：Slave发起的通道（R读数据、B写响应）。
  • 代码写法：Slave内部状态机在没算好数据时，VALID保持为0；一旦算好并寄存到输出端口，立即将VALID置1。它不管Master此刻READY是高是低，只管把数据摆出来"宣告"我准备好了。
  • 效果：解耦。防止慢速Slave拖累快速Master的内部时序，谁准备好了谁先表态。

常见误区

  • 写操作包含了 AW（Master发）、W（Master发）、B（Slave发） 三个通道。其中AW和W因为是Master发，所以用"同时/READY先高"效率高；而B因为是Slave发，依然要用"VALID先高"。
  • 读操作包含了 AR（Master发） 和 R（Slave发）。其中AR因为是Master发，用"同时/READY先高"；R因为是Slave发，必须用"VALID先高"。

十七、AXI4 读事务时序

时序解读：

  • T1→T2：Master拉高ARVALID，Slave在T2周期拉高ARREADY，地址握手在T2上升沿完成。
  • T2→T4：Slave准备数据，在T4周期拉高RVALID，Master在T5周期拉高RREADY，数据握手在T5上升沿完成（首次数据）。
  • 地址通道和数据通道完全独立：地址握手在T2完成，数据握手在T5完成，中间有3拍延迟（这是Slave的读延迟）。
  • 突发传输：RLAST在最后一拍数据时拉高（图中DATA1是最后一拍）。

十八、AXI4 写事务时序

时序解读：

  1. AW和W完全独立：Master在T1发地址，在T4才发数据。Slave在T2接收地址，在T5接收数据。
  2. WLAST：在最后一拍数据时拉高。
  3. B通道：Slave在收到所有数据并写入完成后才拉高BVALID返回响应。BVALID在T6或更晚出现。

十九、AXI4 高级特性

超前传输（Outstanding）

Outstanding 是指Master在等待第一个事务的响应（R或B）之前，就可以继续发送后续事务的地址。

时序解读：

  • Master在T1、T2、T3连续发送3个地址（A0、A1、A2），不需要等待D0返回。
  • Slave可以在后续连续返回数据D0、D1、D2（顺序或乱序，由ID控制）。
  • Outstanding深度 = 3（本例中）。

乱序传输（Out of Order）与交织传输（Interleaving）

  • 乱序传输（Out-of-Order）：

  • 作用对象：不同ID（或者不同通道）的整个突发传输（Burst）。

  • 定义：主机（Manager）发出的多个写/读请求（地址通道AR/AW）的顺序，与数据通道（W/R）上完成传输的顺序，或者响应通道（B）上返回的顺序不一致。

  • 本质：请求顺序的重排。目的是让先发起但被阻塞的慢操作，让位给后发起但准备就绪的快操作。

  • 交织传输（Interleaving）：

  • 作用对象：同一个ID的单个突发传输内部的各个数据节拍（Beats）。

  • 定义：在读取回传数据（R通道）时，来自不同ID的多个突发传输的数据节拍，可以交错地在总线上进行传输。

  • 本质：数据包的分割与穿插。目的是防止某个长突发长时间占据总线，导致其他ID的短突发饿死。

比较维度 乱序传输 （Out-of-Order） 交织传输 （Interleaving）
粒度 Burst 级别（整个突发完成顺序变化） Beat 级别（突发内部单个数据传输顺序变化）
依赖标识 ARID / AWID（事务ID） RID（读数据ID，必须与ARID匹配）
响应通道（B） 允许乱序（BID 顺序与 AWID 不同） 不涉及（写响应没有交织概念，B通道顺序与写数据交织无关）
关键前提 从机必须声明支持 OutOfOrder 能力 从机必须声明支持 Interleaving 能力
限制规则 相同 ID 的突发必须严格保序 相同 ID 的 Beat 必须连续传输（中间不能插入其他ID的Beat）

不同ID的事务可以乱序完成，但同一ID的事务必须严格保序完成。
这条铁律同时适用于读（R通道）和写（B通道），是AXI乱序机制的最高准则。

AXI引入乱序传输机制的核心目的是把调度自由度下放给从机。

读数据乱序

不同 ARID 的读事务，其数据在R通道上可以交织返回，也就意味着后发起的读事务可以先于先发起的完成。

规则重述：

  • 不同 ARID：允许乱序返回
  • 同一 ARID：必须保序返回

读乱序的存在意义在于：从机可以根据内部资源的就绪状态灵活响应，不需要因为先发起的请求被阻塞就饿死后发起的就绪请求。

写数据乱序（AXI4 vs AXI3）

这是AXI4与AXI3的重大区别，也是乱序传输中最容易被误解的地方。AXI4中写通道的乱序行为需要分两个层面来看：

通道 是否允许乱序 原因
写数据（W通道） 不允许 AXI4移除了WID信号，从机只能按物理接收顺序匹配数据与地址
写响应（B通道） 允许 响应自带BID标签，从机可独立返回任何ID的完成状态

⚠️ 容易混淆点： “写通道乱序"≠"写数据乱序”。写数据必须顺序发送，但写响应可以乱序返回。

乱序 + 交织场景示例

假设主机发出了两个读请求（以读操作为例）：

  • ID=0：读地址 Addr_A，突发长度 4 beats（访问慢速外设，响应延迟大）。
  • ID=1：读地址 Addr_B，突发长度 2 beats（访问快速缓存，响应延迟小）。
    发送顺序（地址通道）： Addr_A（ID=0） --> Addr_B（ID=1）

场景一：仅支持乱序，不支持交织（普通高性能模式）

  • 数据通道表现：
  1. ID=0 的数据准备很慢，但 ID=1 的数据已就绪。
  2. 允许乱序：总线先传输 ID=1 的两个 Beat（B1, B2）。
  3. ID=1 传输完毕后，再传输 ID=0 的四个 Beat（A1, A2, A3, A4）。
  • 总线波形（R通道）：B1 -> B2 -> A1 -> A2 -> A3 -> A4
  • 结论：Burst 完成的顺序是 1 先于 0（与请求顺序 0->1 相反），实现了乱序。但每个 Burst 内部的数据是连续且不可分割的。

场景二：支持交织，假设不支持乱序（严格保序模式）

  • 数据通道表现：
  1. 由于不允许乱序，必须等 ID=0 的 Burst 全部传完，才能传 ID=1。
  2. 但 ID=0 准备很慢，且 ID=1 数据已就绪。
  • 总线波形（R通道）：A1（等待很久）-> A2 -> A3 -> A4 -> B1 -> B2
  • 结论：虽然保序，但总线利用率极低（等待A1时，B数据空闲）。

场景三：乱序 + 交织 同时支持（极致性能模式）

  • 数据通道表现：
  1. ID=0 的 Beat 1 (A1) 还没准备好，但 ID=1 的 B1, B2 好了 -> 乱序生效，先发 B1, B2。
  2. 发完 ID=1 的 Burst 后，ID=0 的 A1 依然没准备好。
  3. 此时，如果继续傻等 A1，总线会空闲。但交织介入：由于 ID=1 已经传完，没有其他 ID 可以穿插了？—— 为了体现交织，我们假设 ID=2（短突发）也在等待。
  • 极致情况演绎：
  • 时间片 T1：R通道发送 ID=1 的 Beat 1。
  • 时间片 T2：R通道发送 ID=1 的 Beat 2（ID=1 Burst结束）。
  • 时间片 T3：ID=0 的 A1 还没到，但 ID=2 的数据到了。交织允许在 ID=0 的 Burst 还没结束时，插入 ID=2 的数据。
  • 总线波形（R通道）：B1(ID=1) -> B2(ID=1) -> C1(ID=2) -> C2(ID=2) -> A1(ID=0) -> A2(ID=0) -> A3 -> A4
  • 深度解析：
  • 乱序体现：ID=1 和 ID=2 的 Burst 完成顺序在 ID=0 之前。
  • 交织体现：在 ID=0 这个长突发的时间跨度内，插入了 ID=2 的完整 Burst 数据。

在实际 SOC 设计中，交织虽然能提高吞吐，但会极大增加从机内部的状态机复杂度（需要维护多个 RID 的当前计数值）。因此，AXI4 协议规定，交织深度（Interleaving Depth）必须由从机在 RUSER 或配置参数中声明，且主机在发起传输时，需要确保接收缓冲区能够容纳交织带来的乱序数据重组压力。

二十、AXI4的Outstanding深度控制

Outstanding是AXI实现高带宽的核心特性之一。它打破了传统总线"发起请求→等待响应→再发起请求"的串行模式，允许Master在未收到前一个事务响应时，就连续发出多个新事务的地址。这样，即使访问DDR这类高延迟外设，总线也能在等待期间处理其他请求，从而隐藏访问延迟，大幅提升带宽利用率。

要正确实现Outstanding，关键在于深度控制，这涉及三个层面的协同配合：

Master的发送能力（MAX_OUTSTANDING）

Master内部维护一个计数器，记录当前"在途"（已发地址但未收响应）的事务数量。这个计数器会随着地址握手成功而增加，随着响应完成而减少。

MAX_OUTSTANDING的上限通常取决于Master内部为保存在途事务信息而配备的缓冲区大小，每个在途事务需要缓存地址、控制信息以及后续返回的数据。

Interconnect的转发能力（ISSUE_CAPACITY）

总线互联结构（Interconnect）本身也有处理能力上限，需要能够容纳和转发来自多个Master的Outstanding请求。如果多个Master同时发送大量Outstanding请求，Interconnect内部的缓冲队列可能溢出。

Slave的接收能力（ACCEPT_CAPACITY）

Slave通常有自己的队列来缓存接收到的请求。如果Slave的Outstanding能力为N，意味着它可以同时处理N个未完成的事务。如果Master发送速度过快，超过了Slave的处理能力，Slave可以通过拉低对应的AxREADY信号来进行反压，告诉Master"我暂时处理不了新的请求了"。

二十一、AXI4-Lite Slave RTL代码

AXI4-Lite是AXI的简化版，不支持突发传输（LEN固定为0），但保留了完整的5通道握手。它最适合作为寄存器的配置接口。

AXI4-Lite Slave 代码要点：

  1. 地址和数据通道独立握手：状态机同时监测AWVALID和WVALID，支持地址先到或数据先到。
  2. 写响应延迟：必须等到地址和数据都握手完成后，才进入WR_RESP状态返回BVALID。
  3. WSTRB字节选通：支持部分字节写入，适用于对32位寄存器中个别字节的更新。
  4. 读数据零延迟：使用组合逻辑，ARVALID有效时RDATA在同一周期输出（适用于寄存器读）。

二十三、AXI4-Stream 协议详解

AXI4-Stream 与前面讲的 AXI4 最大的区别是：没有地址通道。它不涉及"读某个地址"或"写某个地址"，只有纯粹的数据流发送与接收。这让它非常适合视频流、音频流、高速ADC/DAC数据采集等场景。

核心信号

信号 方向 描述
ACLK - 全局时钟，上升沿采样
ARESETn - 低电平复位，复位期间TVALID必须为0
TVALID Master→Slave 主机数据有效标志
TREADY Slave→Master 从机准备好接收
TDATA Master→Slave 数据总线（宽度可配置）
TSTRB Master→Slave 字节类型标志（1=数据字节，0=位置字节）
TKEEP Master→Slave 字节有效标志（1=该字节需传输到目的地）
TLAST Master→Slave 当前Packet的最后一个数据
TID Master→Slave 数据流标识符（可选）
TDEST Master→Slave 路由信息（可选）
TUSER Master→Slave 用户自定义信息（可选）

握手机制

AXI4-Stream 的握手与 AXI4 一致，遵循 VALID/READY 双向握手机制：

关键规则：

  1. Master不能等TREADY拉高才拉TVALID。一旦TVALID拉高，必须保持直到握手完成（TVALID && TREADY）。
  2. Slave可以等TVALID拉高再拉TREADY，也可以提前拉TREADY。
  3. TLAST 在Packet最后一个数据时拉高，表示本次流传输结束。

二十五、桥接器设计（Bridge）

在SoC中，高速总线（AXI/AHB）与低速总线（APB）之间需要桥接器来转换协议。桥接器的本质是协议转换状态机。

AXI-to-APB Bridge 架构

AXI-to-APB桥接器的核心功能是将 AXI4-Lite事务 转换为 APB3事务。

工作流程：

  1. AXI读事务：AXI Slave收到AR通道的读地址 → 状态机转换为APB读操作（PSEL=1, PENABLE=1, PWRITE=0）→ 从APB Slave读回PRDATA → 通过R通道返回给AXI Master。
  2. AXI写事务：AXI Slave收到AW通道地址 + W通道数据 → 状态机转换为APB写操作（PSEL=1, PENABLE=1, PWRITE=1, PWDATA=数据）→ 等待APB传输完成 → 通过B通道返回写响应。

地址译码：桥接器内部包含地址译码逻辑，将AXI地址映射到对应的APB Slave的PSEL信号。

AHB-to-APB Bridge 核心状态机

AHB-to-APB桥接器是AHB总线上的一个Slave，将AHB的高速事务转换为APB事务。

状态跳转：

  • IDLE：等待AHB传输请求（HTRANS=NONSEQ）。
  • ENABLE：拉高PSEL和PENABLE，启动APB传输。如果PREADY=0，进入WAIT状态。
  • WAIT：等待PREADY拉高。拉高后完成传输，回到IDLE。

AHB与APB的时钟关系：如果AHB和APB时钟频率相同，APB的PRDATA可以直接赋值给AHB的HRDATA。如果APB时钟更慢，则需要寄存器缓存读数据，等APB完成后在下一拍返回给AHB。

二十六、CHI协议详解

CHI（Coherent Hub Interface）是AMBA 5引入的新一代一致性互联接口。如果说AXI是高效灵活的"高速公路"，那CHI就是为大规模、多核CPU设计的"立体交通网"。它的一个重要设计思想就是分层，将协议层、网络层和链路层分开，每层各司其职。

CHI的分层架构

这种分层设计的好处在于：链路层可以专注于在物理链路上高效、可靠地传输数据，而无需关心具体的事务含义。同时，各层可以独立优化和迭代。

CHI链路层核心概念

链路层的核心工作，可以概括为以下几个关键概念：

① Flit（流控单元）：链路层的"数据包"

CHI链路层不再直接传输协议中的"数据包"或"事务"，而是将其拆分成更小的、固定格式的传输单元，称为Flit (Flow Control Unit)。

Flit类型 用途 说明
Protocol Flit 传输协议事务 主力军。一个CHI协议层的数据包（Packet）被封装进一个Protocol Flit里，承载实际的事务信息，如读请求、写数据、监听请求等
Link Flit 链路层管理 勤务兵。它不传输业务数据，而是用于链路自身的维护和管理，最重要的用途就是传递链路层信用（L-Credit）

② 通道（Channel）：分工明确的"车道"

CHI链路层定义了四类通道，不同类型的Flit各行其道。这有点像AXI分离地址、数据通道的思想，但在CHI中分得更细，并且增加了"监听"这一关键通道来维护缓存一致性。

通道 方向 传输内容 类比AXI
请求(REQ) 请求节点→主节点 读/写请求、一致性操作请求 AR/AW通道
响应(RSP) 主节点→请求节点 一致性响应、数据响应 R/B通道
监听(SNP) 主节点→请求节点 缓存监听请求、一致性探测 无对应（ACE新增）
数据(DAT) 双向 实际读写数据 W/R数据通道

③ 链路层信用（L-Credit）：从"你发我收"到"按信用借贷"

这是CHI链路层最核心的流控机制，与AXI的READY/VALID反压有本质不同。它不再是简单的"收方告诉我’准备好’了再发"，而是采用了基于信用的流控：

  • 信用借贷：链路接收方（Rx）会通过Link Flit告诉发送方（Tx），它有多少可用的接收缓冲区（即信用，Credit）。

  • 先发后查：Tx端维护一个信用计数器。每发送一个Flit，就消耗一个信用；每收到一个来自Rx的L-Credit，信用就增加。只要信用足够，Tx无需等待对方实时握手就可以连续发送数据，避免了每次传输的握手延迟，实现了更高效率的流水线。

从实际芯片（如"香山"处理器）的接口信号中也能看到这一机制的痕迹，例如chi_tx_req_lcrdv这个输入信号，就是用来接收对端返回的L-Credit的。

④ 通道依赖关系：避免死锁的"交通规则"

为了确保系统不会因为等待而卡死，CHI对各个通道之间的依赖关系做了严格限制。

这些规则就像十字路口的交通规则，看似繁琐，但它们是保证整个复杂系统能够正常流转起来的基础。

ACE vs CHI：关键差异

特性 ACE CHI
信号基础 扩展的AXI信号线 完全重构，基于数据包(Flit)
流控机制 VALID/READY握手（反压） 基于信用(Credit)的流控
通道数 5个扩展通道 4个链路层通道(REQ/RSP/SNP/DAT)
可扩展性 适用于16核心以下 支持百核级别
协议层次 扁平化 协议层/网络层/链路层三层架构
链路效率 每拍握手有开销 信用机制允许连续发送，效率更高

二十七、验证方法学

基于UVM的验证

对于AMBA协议的验证，业界主流方法是UVM（Universal Verification Methodology） + SystemVerilog。一个典型的UVM验证环境包含：

组件 功能
Driver 将测试用例中的事务（Transaction）驱动到DUT的接口上
Monitor 监视总线活动，采集信号并打包成事务
Scoreboard 对比预期结果和实际结果，检查数据完整性
Coverage 收集功能覆盖率，确保所有场景都被测试到

UVM验证APB的核心检查点：

  • 协议合规性：PSEL/PENABLE/PREADY的时序关系是否符合规范。
  • 数据完整性：读/写数据是否正确。
  • 错误处理：PSLVERR在非法访问时是否正确拉高。
  • 边界条件：背靠背传输、等待插入、复位下的行为。

二十八、高频问答题汇总（附答案）

AHB相关

  1. AHB有哪些组成部分？ → 主机(Master)、从机(Slave)、仲裁器(Arbiter)、译码器(Decoder)。

  2. burst传输能跨越1K边界吗？为什么？ → 不能。AHB规定burst不能跨越1KB边界，因为Slave的地址译码通常以1KB为单位，跨越边界会导致译码错误。

  3. AHB slave短时间无法响应怎么办？ → 拉低HREADY插入等待周期。

  4. AHB slave长时间无法响应怎么办？ → 返回SPLIT或RETRY响应，释放总线让其他Master使用。

  5. HTRANS中的IDLE和BUSY有什么区别？ → IDLE插在两次burst之间，表示Master不发起传输；BUSY插在同一个burst中间，表示Master暂时无法提供数据/地址，但burst还会继续。

APB相关

  1. APB传输有什么特点？ → 非流水线，至少2个周期（SETUP + ACCESS），最快2拍完成一次传输。

  2. PSLVERR在什么情况下拉高？ → 读只写地址、写只读地址、访问未定义地址时，Slave拉高PSLVERR表示错误。

AXI相关

  1. AXI的5个通道是什么？ → 读地址(AR)、读数据®、写地址(AW)、写数据(W)、写响应(B)。

  2. Outstanding传输是什么？ → Master在等待第一个事务的响应之前，可以继续发送后续事务的地址。

  3. AXI如何支持乱序响应？ → 通过ID信号。Slave返回数据时带上RID，Master根据ID匹配对应的事务。

  4. AXI的Outstanding深度由什么决定？ → 三方面：Master的缓冲区大小、Interconnect的队列深度、Slave的接收能力。任何一个环节的瓶颈都会限制实际有效的Outstanding深度。

  5. AXI4中写数据通道是否支持乱序？为什么？ → 不支持。AXI4移除了WID信号，从机只能按物理接收顺序匹配数据与地址。但写响应（B通道）可以通过BID标签实现乱序返回。

  6. AXI的FIXED突发类型适用于什么场景？ → 适用于反复读写同一个地址的场景，比如读写FIFO的同一个端口。

  7. AXI的WRAP突发类型与INCR有什么区别？ → WRAP用于Cache Line填充，地址到达边界后会回卷；INCR则一直递增直到突发结束。

CHI相关

  1. CHI相比ACE的主要优势是什么？ → CHI采用三层架构（协议/网络/链路），基于Flit和Credit的通信机制，支持更高的可扩展性和链路效率。

  2. CHI的链路层信用机制如何工作？ → 接收方通过Link Flit告知发送方可用缓冲区数量（信用），发送方维护信用计数器，有信用则可连续发送，无需每拍握手。

  3. CHI通道依赖关系的核心规则是什么？ → 请求节点可以收到监听后才发数据响应，但不能发请求后等待监听，否则可能导致死锁。

  4. CHI中的SNP通道有什么作用？ → SNP（Snoop）通道用于缓存一致性监听请求，主节点通过SNP通道向请求节点发起一致性探测，确保多核缓存数据的一致性。
