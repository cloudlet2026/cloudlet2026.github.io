第一章：GMAC概述与协议架构

1.1 什么是GMAC

GMAC（Gigabit Media Access Controller，千兆媒体访问控制器）是遵循IEEE 802.3标准实现的以太网MAC层控制器，主要负责数据链路层的介质访问控制功能。在嵌入式系统中，GMAC通常作为SoC内部集成的硬件外设，承担与外部PHY芯片通信、以太网帧的封装与解封装、DMA数据传输等核心任务。

从OSI七层模型来看，GMAC工作在数据链路层的下半部分，向上对接网络层的IP协议栈（如lwIP），向下通过MII/GMII/RGMII等接口连接物理层PHY芯片。其典型应用场景包括工业控制、网络设备、智能终端及物联网边缘节点等对高带宽、低延迟有严苛要求的领域。

1.2 GMAC与PHY的关系

在实际硬件设计中，由于PHY包含大量模拟器件，而MAC是典型的数字电路，考虑到芯片面积及模数混合架构的复杂性，通常将MAC集成进SoC，将PHY留在片外。两者之间通过标准接口互联：

MAC的作用：对物理层送上来的数据进行区分——如果数据包目的地址是本机，则处理后提交上层协议；否则丢弃。发送时，在待发送数据上添加前导码、帧首定界符、FCS等信息后交给PHY。

PHY的作用：实现物理层功能，包括数字信号与模拟信号的转换、速度协商、载波侦听与冲突检测等。PHY还通过Auto-Negotiation技术与对端设备协商连接速度、双工模式和流控策略。

一个GMAC可以同时管理多个PHY芯片，通过MDIO接口进行寄存器读写和状态监控。

1.3 GMAC的核心功能模块

GMAC硬件通常划分为以下四个主要部分：

模块 功能描述
帧发送（Frame Transmission） 接收上层数据，添加前导码、SFD、FCS，组成以太网帧发送
帧接收（Frame Reception） 接收PHY传来的位流，检查帧有效性，去除帧头尾提交上层
GMAC控制（GMAC Control） 全双工模式下控制帧（Pause Frame）的生成与检测
MII管理接口（MDIO） 控制PHY的寄存器读写，检查PHY状态

1.4 GMAC工作模式

半双工模式：发送前需载波侦听（CRS），检测到载波则等待。发送过程中若检测到冲突（Collision），根据冲突类型（normal/late）执行jam序列发送和退避重传，重传超过16次则丢弃数据包。

全双工模式：无需载波侦听和冲突检测，可直接发送数据，同时支持Pause Frame流控。

第二章：MII/GMII/RGMII接口详解

2.1 接口体系概览

MAC与PHY之间的连接接口统称为介质独立接口（Media Independent Interface），随着以太网速率提升，衍生出多个变种：

接口 数据位宽 时钟频率 速率支持 引脚数
MII 4-bit 25MHz/2.5MHz 100M/10M ~16
GMII 8-bit 125MHz 1000M ~24
RMII 2-bit 50MHz 100M/10M ~10
RGMII 4-bit DDR 125MHz(DDR) 1000M ~12
SGMII 串行 625MHz DDR 1000M 4

2.2 GMII接口信号

GMII（Gigabit MII）是千兆以太网的标准接口，包含以下关键信号：

数据通道：

  • GTX_CLK：125MHz发送时钟（MAC输出）
  • TX_CLK：接收时钟（PHY输出，10/100M模式）
  • TXD[7:0]：8位发送数据
  • RXD[7:0]：8位接收数据
  • TX_ER：发送错误指示
  • RX_ER：接收错误指示
  • TX_EN：发送使能
  • RX_DV：接收数据有效
  • CRS：载波侦听
  • COL：冲突检测

2.3 RGMII接口与时序

RGMII（Reduced GMII）通过DDR（双沿采样）技术将引脚数从24个减少到12个左右，是目前嵌入式SoC最常用的GMAC-PHY连接方式。

RGMII的核心特性：

  • TXD/RXD数据线从8-bit缩减为4-bit
  • 时钟频率仍为125MHz，但数据在时钟上升沿和下降沿同时采样，实现250Mbps/线
  • 4线总计1000Mbps传输速率

⚠️ 时序注意事项：由于RGMII在时钟双沿采样，对信号延迟敏感。实际设计中通常需要PCB走线等长处理，并根据PHY型号调整TX/RX时钟的相位延迟（通过MAC或PHY内部的delay配置）。

第三章：MDIO管理接口与PHY访问

3.1 MDIO协议简介

MDIO（Management Data Input/Output）是IEEE 802.3定义的管理接口，用于MAC访问PHY的内部寄存器，实现配置、状态读取、自协商控制等功能。

MDIO接口包含两根信号线：

  • MDC：管理时钟，由MAC输出，最高频率通常为2.5MHz
  • MDIO：双向数据线，用于传输读写命令和数据

3.2 MDIO帧格式

MDIO协议帧格式如下：

  • 前导码：32个连续"1"，用于同步
  • 起始码："01"表示帧开始
  • 操作码：“01”=写，“10”=读
  • TA（Turn Around）：读操作时为"Z0"（PHY驱动数据），写操作时为"10"

3.3 PHY地址与扫描

每个PHY芯片在MDIO总线上有4位地址（0-31），地址0通常作为广播地址。部分PHY芯片在上电或复位时根据硬件引脚（PHYAD[2:0]）设置地址的低3位。

地址扫描流程：若已知PHY地址无效，可遍历0-31地址，读取PHYID1/PHYID2寄存器，与目标PHY型号的ID值比对来确认正确的PHY地址。

第四章：GMAC DMA与描述符机制

4.1 DMA架构概述

GMAC通常集成专用的DMA控制器，用于在内存和MAC之间自动搬运数据，无需CPU干预。DMA基于描述符环（Descriptor Ring） 机制工作——这是一个在内存中连续存储的描述符数组，每个描述符指向一个数据缓冲区。

4.2 发送描述符（TX Descriptor）

发送描述符定义了待发送数据帧在内存中的位置、长度和控制信息。典型的发送描述符结构（以STM32/瑞萨GMAC为例）：

TDES0关键位域：

  • OWN（bit 31）：DMA拥有权标志。1表示DMA可访问，0表示CPU可访问
  • FS（bit 28）：帧首段标志。表示该描述符是数据包的起始
  • LS（bit 29）：帧尾段标志。表示该描述符是数据包的结尾
  • IC（bit 30）：传输完成中断标志。LS包时设置此位触发中断
  • TTSE（bit 25）：发送时间戳使能（用于IEEE 1588）

TDES1位域：

  • 高16位：缓冲区2长度
  • 低16位：缓冲区1长度

4.3 接收描述符（RX Descriptor）

接收描述符定义DMA将数据存入内存的位置：

RDES0关键信息：

  • OWN（bit 31）：DMA拥有权
  • FL（bits 29:16）：帧长度
  • FS/LS：帧首/帧尾标志
  • CRC/ER：CRC错误/接收错误
  • MCH：多播帧标志
  • BC：广播帧标志

4.4 多描述符传输大包

默认配置下，每个数据包使用一个描述符，每个缓冲区大小通常为2048字节。对于内存受限的系统，可以将缓冲区减小到256字节，让大包跨越多个描述符：

大包（900字节）使用4个描述符：

  • 第1个描述符：FS=1, LS=0 → 数据包起始
  • 第2个描述符：FS=0, LS=0 → 数据包中间
  • 第3个描述符：FS=0, LS=0 → 数据包中间
  • 第4个描述符：FS=0, LS=1, IC=1 → 数据包结束，触发完成中断

小包（200字节）使用1个描述符：

  • 第1个描述符：FS=1, LS=1, IC=1 → 同时是起始和结束

第五章：GMAC数据发送与接收完整流程

5.1 发送数据流程详解

发送操作的核心是CPU准备好数据帧后，将描述符交给DMA，由DMA自动完成从内存到MAC的传输。完整流程分为六个阶段：

阶段一：CPU填充数据缓冲区

阶段二：配置描述符控制信息

阶段三：将描述符所有权交给DMA

阶段四：触发DMA发送

阶段五：硬件自动处理
DMA检测到OWN=1的描述符后，依次执行：

  1. 从描述符TDES2指向的内存读取数据
  2. 将数据送入MAC的发送FIFO
  3. MAC添加前导码（7字节0x55）、SFD（0xAB）、FCS（CRC32）
  4. 通过GMII/RGMII接口向PHY发送数据流

阶段六：发送完成与中断处理
当DMA完成数据发送且描述符设置了IC位，硬件会置位发送完成中断标志。中断服务程序（ISR）中需清除OWN位，释放描述符供后续使用：

5.2 接收数据流程详解

接收流程由硬件主导，当PHY收到数据后，MAC剥离前导码和FCS，将有效数据通过DMA写入接收描述符指向的缓冲区，并触发中断。

阶段一：硬件接收与DMA写入

  1. PHY接收到以太网帧，通过RGMII向MAC发送数据

  3. 数据通过DMA写入当前OWN=1的接收描述符缓冲区
  4. 接收完成后，RDES0的FL字段记录帧长度，OWN位被硬件清零

第六章：中断系统与事件处理

6.1 GMAC中断源与优先级设计

GMAC通常支持以下中断源，需要通过厂商提供的库函数或直接配置寄存器来使能：

中断源 标志位 触发条件 推荐优先级
接收完成 RX_COMP 接收到完整帧 最高
发送完成 TX_COMP 帧发送完成（IC位使能） 中高
接收错误 RX_ERR CRC错误、对齐错误等 中
发送错误 TX_ERR 冲突重试超限等 中
接收溢出 RX_OVR 描述符耗尽导致丢帧 中
链路变化 LINK_CHG PHY链路状态变化 低

优先级设计原则：

  • 接收中断优先级最高，防止FIFO溢出丢包
  • 发送中断次之，确保CPU及时回收描述符
  • 错误/状态中断放在较低优先级，可通过轮询方式处理

第八章：GMAC高级功能与配置

8.1 巨型帧（Jumbo Frame）支持

标准以太网帧最大1518字节（含头部和FCS），巨型帧支持更大的MTU（通常可达9000字节），提升大数据量传输效率。

配置步骤：

  1. 在LwIP中设置更大的MTU：netif->mtu = 9000;
  2. GMAC寄存器中启用巨型帧模式（MTU寄存器设值>1518）
  3. 接收缓冲区增大到对应大小（如9216字节）
  4. PHY需支持巨型帧（无需特殊配置，由MAC处理）

高端GMAC支持硬件时间戳，用于实现纳秒级时钟同步。

第十章：常见问题与FAQs

Q1：PHY无法识别/读取ID全是0xFFFF或0x0000

可能原因：

  • 硬件引脚连接错误（MDC/MDIO反接）
  • PHY地址错误（地址0通常为广播地址，应避开）
  • MDC时钟频率过高（建议起始设为2.5MHz/2分频）
  • PHY芯片未上电或复位引脚电平不正确

排查步骤：

  1. 用示波器测量MDC和MDIO信号，确认有通讯波形
  2. 遍历PHY地址1-31，读取PHYID1/PHYID2寄存器
  3. 检查PHY的复位时序：复位后需等待足够时间（典型10-50ms）
  4. 查看PHY数据手册中的默认地址配置（引脚拉高/拉低）

Q2：能接收数据但无法发送/发送中断不触发

检查清单：

  1. 确认发送描述符的OWN位已正确置1
  2. 确认GMAC_NCR中的TXEN位已使能
  3. 检查TX_CLK是否存在（某些SoC需要配置引脚复用功能）
  4. 半双工模式下检查是否有持续的载波冲突（观察COL引脚）
  5. 确认描述符中的FS/LS标志正确设置——只有设置LS的帧才会触发发送完成中断

Q3：接收数据偶尔丢包或CRC错误率偏高

排查方向：

  • 信号完整性：检查RGMII走线是否等长，建议长度差<5mm
  • 时钟相位：尝试切换GMAC内部的TX/RX时钟延迟配置（调试方法见下文）
  • 电源噪声：PHY的模拟电源（AVDD）需使用LDO单独供电
  • 接地：确保PHY的AGND和DGND在单点连接

Q5：MAC地址应该从哪里获取？

推荐方案（按优先级排列）：

  1. 芯片唯一ID：使用MCU的96位唯一ID（如STM32的UID）计算生成
  2. OTP/eFuse：量产时在芯片的一次性可编程区域写入
  3. 外部EEPROM：板载存储MAC地址
  4. 默认值+随机：开发阶段可临时使用默认值，配合随机化避免局域网冲突

Q6：半双工模式下工作不稳定

半双工模式对时序和信号完整性要求更高。建议：

  1. 确认PHY协商结果确实是半双工（读取PHY_ANLPAR寄存器）
  2. 检查冲突计数器是否异常增加
  3. 若非必须，强烈建议改用全双工模式
  4. 如需半双工，减小网络总长度，使用高质量双绞线

Q7：如何实现多GMAC实例（双网口）？

大部分SoC支持2个GMAC实例，要点：

  1. 为每个实例分配独立的描述符环和缓冲区
  2. 使用不同的中断向量（如GMAC0_IRQn和GMAC1_IRQn）
  3. 为每个接口创建独立的struct netif
  4. 确保两个GMAC的PHY地址不同
  5. 注意总线带宽——双千兆全速时，DMA带宽可能成为瓶颈

Q8：复位后需等待多久才能访问PHY？

典型时序要求：

  • PHY上电后：需等待10-50ms（内部模拟电路稳定）
  • PHY软复位后：需等待约100-500μs（数据手册有具体值）
  • 自协商完成：通常1-3秒（最坏可达5秒）
  • 建议在自协商等待期间循环读取BMSR，超时则回退到强制模式
