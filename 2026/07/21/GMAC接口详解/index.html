第一章：GMAC 概述与协议架构

1.1 什么是 GMAC

GMAC（Gigabit Media Access Controller，千兆媒体访问控制器）是遵循 IEEE 802.3-2018 标准实现的以太网 MAC（Media Access Control）层控制器。它负责数据链路层的介质访问控制功能，包括帧的封装与解封装、错误检测、流量控制等。

在典型的嵌入式SoC（System-on-Chip）中，GMAC作为一个硬件外设集成在芯片内部，通过标准总线（如AXI、AHB）与CPU和内存相连，对外则通过MII/GMII/RGMII等接口连接物理层PHY芯片。

从OSI模型来看：

  • 上层（网络层）：与IP协议栈（如lwIP、uC/TCP-IP）交互，收发IP报文。
  • 下层（物理层）：通过MII/GMII/RGMII等接口与PHY芯片通信，管理数据的物理传输。

1.2 GMAC 与 PHY 的关系

在实际硬件设计中，由于PHY包含大量模拟电路（如ADC/DAC、PLL），而MAC是纯数字逻辑，因此通常将MAC集成在SoC内，PHY作为独立芯片或片外模块。两者之间通过标准化的介质独立接口互联。

GMAC（MAC侧）职责：

  • 发送：从内存（通过DMA）读取帧数据，添加前导码（Preamble）、帧首定界符（SFD）和帧校验序列（FCS），然后通过接口发送给PHY。
  • 接收：从PHY接收数据流，剥离前导码和FCS，校验帧正确性，将有效负载通过DMA写入内存。
  • 管理：通过MDIO接口读写PHY寄存器，进行配置和状态监控。
  • 流控：在全双工模式下生成和处理Pause帧。

PHY（物理层）职责：

  • 将MAC发送的数字信号（如RGMII的4位DDR数据）转换为适合线缆传输的模拟信号（如1000BASE-T的PAM5编码）。
  • 将线缆接收的模拟信号转换为数字信号。
  • 自动协商（Auto-Negotiation）：与对端设备协商速率（10/100/1000Mbps）、双工模式（半双工/全双工）和流控能力。
  • 链路状态监测（Link Status）。

1.3 GMAC 核心功能模块

一个典型的GMAC硬件内部包含以下主要功能块：

模块 功能描述
MAC发送控制器 从MTL（MAC Transaction Layer）FIFO读取数据，添加前导码、SFD和CRC，控制发送时序。
MAC接收控制器 从接口接收数据流，检测前导码、SFD，校验CRC，将有效帧写入MTL接收FIFO。
MTL（MAC Transaction Layer） 提供发送/接收FIFO，进行数据缓冲和速率匹配。也负责流量控制（Pause帧生成）。
DMA控制器 在系统内存和MTL FIFO之间搬运数据，基于描述符环（Descriptor Ring）工作。
控制与状态寄存器（CSR） 提供软件配置接口（速率、双工、滤波、中断使能等）和状态查询。
MDIO（管理接口） 实现IEEE 802.3 Clause 22/45管理帧协议，用于访问PHY寄存器。
中断控制器 汇集并上报各种事件（收发完成、错误、链路变化等）至CPU。

1.4 GMAC 工作模式

  • 半双工模式（Half-Duplex）：仅用于10/100Mbps共享介质环境（如集线器）。发送前需进行载波侦听（CSMA/CD），发送过程中若检测到冲突（COL），则执行退避算法并重传。重传超过16次则丢弃帧。
  • 全双工模式（Full-Duplex）：用于交换式以太网，可同时收发，无冲突检测。支持基于Pause帧的流量控制。

建议：现代嵌入式系统通常使用全双工模式，并优先选择1000Mbps速率。除非有特殊兼容需求，否则不建议使用半双工。

第二章：MII/GMII/RGMII 接口详解

2.1 介质独立接口体系

IEEE 802.3 定义了一系列MAC与PHY之间的连接接口，统称为介质独立接口（Media Independent Interface）。随着速率提升和引脚数优化，衍生出多个变种。

接口 数据位宽 时钟频率 最高速率 引脚数（约） 特点
MII 4-bit 25MHz（100M）/2.5MHz（10M） 100Mbps 16 最经典，广泛支持
GMII 8-bit 125MHz 1000Mbps 24 千兆标准，引脚多
RMII 2-bit 50MHz 100Mbps 10 引脚减少，需50MHz外部时钟
RGMII 4-bit DDR 125MHz（DDR） 1000Mbps 12 引脚少，当前主流
SGMII 串行（1-bit） 625MHz DDR 1000Mbps 4 高速串行，需SerDes

2.2 GMII 接口信号（全功能）

GMII是千兆以太网的原始标准接口，信号定义如下：

数据与时钟：

  • GTX_CLK：125MHz发送时钟，由MAC提供。
  • TX_CLK：接收时钟，由PHY提供（在10/100M模式下，PHY提供25MHz或2.5MHz时钟）。
  • TXD[7:0]：8位发送数据，与GTX_CLK同步。
  • RXD[7:0]：8位接收数据，与TX_CLK同步。

控制信号：

  • TX_EN：发送使能，高有效。
  • TX_ER：发送错误指示，在高电平期间发送无效数据。
  • RX_DV：接收数据有效，高表示RXD[7:0]上的数据有效。
  • RX_ER：接收错误指示。
  • CRS：载波侦听（半双工）。
  • COL：冲突检测（半双工）。

2.3 RGMII 接口与时序

RGMII（Reduced GMII）通过**双沿采样（DDR）**技术，将GMII的8位数据线缩减为4位，同时保持125MHz时钟频率，实现千兆速率。

信号定义：

  • RGMII_TXC：发送参考时钟，125MHz，由MAC提供。
  • RGMII_TXD[3:0]：发送数据，在TXC的上升沿和下降沿各传输4位，组合成8位。
  • RGMII_TX_CTL：发送控制信号，在上升沿为TX_EN，下降沿为TX_EN XOR TX_ER。
  • RGMII_RXC：接收参考时钟，125MHz，由PHY提供。
  • RGMII_RXD[3:0]：接收数据，双沿采样。
  • RGMII_RX_CTL：接收控制信号，编码方式同TX。

⚠️ 时序注意事项：

  • RGMII对信号延迟极度敏感。由于数据在时钟双沿采样，数据和时钟之间的相位偏差必须严格控制。
  • 为解决PCB走线延迟和芯片内部延迟，通常需要在MAC或PHY内部引入可编程延迟（Delay），常见值为1.5~2.0ns（约时钟周期的1/4）。
  • 典型设计有两种方式：
    1. MAC内部延迟：通过MAC寄存器配置TX/RX时钟延迟。
    2. PHY内部延迟：通过PHY寄存器或硬件引脚配置延迟。
  • PCB要求：RGMII信号组需等长走线，长度差建议控制在±5mm以内。

第三章：MDIO 管理接口与 PHY 访问

3.1 MDIO 协议简介

MDIO（Management Data Input/Output）是IEEE 802.3 Clause 22定义的管理总线，用于MAC访问PHY的内部寄存器，实现配置、状态读取、自协商控制及异常诊断。

信号线：

  • MDC：管理时钟，由MAC输出，最高频率通常为2.5MHz（也有支持更高频率的）。
  • MDIO：双向数据线，半双工通信。

3.2 MDIO 帧格式（Clause 22）

MDIO每帧包含以下字段（从MAC角度看）：

字段 位数 值 描述
IDLE - - 空闲时MDIO为高阻
PREAMBLE 32 连续1 帧前导码，用于同步
START 2 01 帧起始标志
OPCODE 2 10（读）/01（写） 操作码
PHY_ADDR 5 0~31 PHY设备地址
REG_ADDR 5 0~31 PHY寄存器地址
TA 2 10（写）/Z0（读） 转向时间（Turnaround）
DATA 16 任意 数据（读操作由PHY驱动，写操作由MAC驱动）

注意：地址0通常作为广播地址，可用于同时访问多个PHY，但实际中很少使用。

3.3 PHY 地址与扫描

每个PHY在MDIO总线上拥有一个5-bit地址（0-31），通常由硬件引脚（如PHYAD[4:0]）在上电复位时锁存。若引脚不足，则使用低3位或4位，高位由内部寄存器默认值决定。

地址扫描流程（若PHY地址未知）：

  1. 遍历地址0~31。
  2. 对每个地址，尝试读取PHYID1寄存器（地址0x02）和PHYID2寄存器（地址0x03）。
  3. 将读回的ID与已知PHY型号的OUI（组织唯一标识符）比对，以确认正确的PHY地址。

第四章：GMAC DMA 与描述符机制

4.1 DMA 架构概述

GMAC内部集成的DMA控制器基于**描述符环（Descriptor Ring）**工作。描述符环是位于系统主存中的一段连续内存区域，其中每个描述符包含指向数据缓冲区的指针、控制标志和状态信息。

核心优势：

  • CPU无需参与数据搬运，只需初始化描述符环并响应中断
  • 支持多帧流水线操作，发送和接收可同时进行
  • 支持帧分割（Scatter-Gather），大帧可跨越多个描述符

4.2 整体架构视角：谁放在哪

在GMAC的DMA数据传输体系中，所有数据和控制结构都存放在**系统主存（通常是DDR/SRAM）**中，但根据功能和访问者不同，分为三个独立的区域：

内存区域 存放内容 访问者 生命周期
描述符环（Descriptor Ring） 描述符数组（控制/状态信息） CPU（写控制）+ DMA（读写状态） 驱动初始化时分配，运行期间持续复用
发送缓冲区池（TX Buffer Pool） 待发送的以太网帧数据 CPU（写数据）+ DMA（读数据） 每个数据包动态使用，发送完成后回收
接收缓冲区池（RX Buffer Pool） 已接收的以太网帧数据 DMA（写数据）+ CPU（读数据） 预先分配固定大小缓冲区，循环使用

4.3 发送描述符（TX Descriptor）

发送描述符定义了待发送数据帧在内存中的位置、长度和控制信息。标准结构如下（以STM32/瑞萨GMAC为例，16字节）：

TDES0 关键位域：

  • OWN（bit 31）：1表示DMA拥有该描述符，0表示CPU拥有
  • FS（bit 28）：帧首段（First Segment）
  • LS（bit 29）：帧尾段（Last Segment）
  • IC（bit 30）：完成中断使能（当LS为1时，发送完成触发中断）
  • TTSE（bit 25）：发送时间戳使能（用于IEEE 1588）
  • WRAP（bit 24）：环回标志（最后一个描述符置1，指示DMA跳到环首）

TDES1：

  • 高16位：缓冲区2长度（以字节为单位）
  • 低16位：缓冲区1长度

若使用双缓冲区模式，则两个缓冲区拼接成一个连续的数据块。若使用单缓冲区，则缓冲区2长度设为0。

4.4 接收描述符（RX Descriptor）

接收描述符定义DMA将接收到的帧数据存放到哪个内存位置。

RDES0 关键信息：

  • OWN（bit 31）：1表示DMA拥有，0表示CPU拥有
  • FL（bits 29:16）：帧长度（有效字节数）
  • FS/LS：帧首/尾标志
  • CRC_ERR（bit 8）：CRC错误
  • RX_ERR（bit 7）：接收错误（如对齐错误、长度错误）
  • MCH（bit 6）：多播帧
  • BC（bit 5）：广播帧
  • WRAP（bit 24）：环回标志

4.5 单描述符与多描述符传输

  • 单描述符：一个帧正好放在一个描述符的缓冲区内。设置 FS=1, LS=1。
  • 多描述符（分片）：大帧分成多个片段，每个片段使用一个描述符。
    • 首片段：FS=1, LS=0
    • 中间片段：FS=0, LS=0
    • 尾片段：FS=0, LS=1, IC=1（触发中断）

大包（900字节）使用4个描述符：

  • 第1个描述符：FS=1, LS=0 → 数据包起始
  • 第2个描述符：FS=0, LS=0 → 数据包中间
  • 第3个描述符：FS=0, LS=0 → 数据包中间
  • 第4个描述符：FS=0, LS=1, IC=1 → 数据包结束，触发完成中断

小包（200字节）使用1个描述符：

  • 第1个描述符：FS=1, LS=1, IC=1 → 同时是起始和结束

4.6 数据缓冲区的分配策略

4.6.1 接收缓冲区池（固定分配）

接收缓冲区在系统初始化时一次性分配，每个缓冲区大小固定（通常为1536字节，足以容纳标准以太网帧）：

内存占用：32 × 1536 = 49,152字节，约48KB。

4.6.2 发送缓冲区（动态使用）

发送缓冲区通常不预先分配固定池，而是由上层协议栈（如LwIP）动态提供。有两种策略：

策略一：零拷贝（Zero-Copy）

  • 协议栈的pbuf数据直接放在描述符指向的地址，无需额外拷贝
  • 要求协议栈分配的内存对DMA可见（物理地址连续）
  • 优点：省去一次内存拷贝，CPU开销小
  • 缺点：对内存管理要求高，协议栈需支持

策略二：拷贝发送（Copy Mode）

  • 驱动预分配一组发送缓冲区（类似接收池）
  • 协议栈数据先拷贝到驱动缓冲区，再交给DMA
  • 优点：实现简单，协议栈无关
  • 缺点：多一次内存拷贝，降低吞吐

4.7 描述符环初始化与配置

描述符环必须位于系统主存中，且首地址对齐到16字节边界（DMA要求）。

4.8 内存布局配置建议

配置项 推荐值 说明
描述符环数量 16~64 太少容易丢包，太多浪费内存
描述符对齐 16字节 满足DMA对齐要求
缓冲区对齐 32字节 Cache Line对齐
缓冲区大小 1536/2048 标准以太网帧+余量

数据缓冲区内存区域 非Cache或Cacheable 同上

如果使用带Cache的Cortex-M7，建议将描述符和数据缓冲区放在**不缓存（Non-cacheable）**的内存区域（如TCM或通过MPU配置），这样无需手动Clean/Invalidate操作，简化驱动逻辑，但会牺牲一些CPU访问速度。如果放在Cacheable区域，则必须严格执行Cache维护操作，否则会出现数据不一致问题。

第五章：数据收发完整流程与数据通路

本章详细描述从软件触发发送到中断处理完成的全流程，DMA如何与CPU协同工作，以及数据在系统中的完整走向。

5.1 CPU与DMA的协同机制概览

在深入流程之前，先理解CPU和DMA各自的角色分工：

机制 CPU角色 DMA角色 同步方式
描述符管理 配置描述符、置OWN=1 读取描述符、清OWN=0 OWN位作为"锁"
数据搬运 准备/消费数据 执行搬运 中断通知完成
中断处理 响应中断、回收/提交 触发中断 中断向量
环形缓冲 维护生产者/消费者指针 内部维护当前指针 指针+OWN位
Cache一致性 Clean/Invalidate操作 直接访问主存 软件保证

关键原则：

  1. 所有权分离：描述符的OWN位是CPU和DMA之间的"令牌"，任何时候只有一个持有者
  2. 生产者-消费者模型：CPU是描述符的"生产者"（发送）或"消费者"（接收），DMA是"消费者"（发送）或"生产者"（接收）
  3. 内存屏障：在置位/清除OWN位前后必须使用内存屏障，防止指令重排导致竞争
  4. Cache维护：在非Cache一致性系统中，必须手动维护Cache与主存的一致性

接收流程由硬件主导，当PHY收到数据后，MAC剥离前导码和FCS，将有效数据通过DMA写入接收描述符指向的缓冲区，并触发中断。

第六章：中断系统与事件处理

6.1 GMAC 中断源概览

中断源 标志位 触发条件 优先级建议
接收完成 RX_COMP 一个完整帧被DMA写入内存 最高
发送完成 TX_COMP 一个帧发送完成（且IC=1） 高
接收溢出 RX_OVR 所有接收描述符被占用，新帧丢弃 中高
发送错误 TX_ERR 冲突重试超限、下溢等 中

第八章：GMAC高级功能与配置

8.1 巨型帧（Jumbo Frame）支持

标准以太网帧最大1518字节（含头部和FCS），巨型帧支持更大的MTU（通常可达9000字节），提升大数据量传输效率。

配置步骤：

  1. 在LwIP中设置更大的MTU：netif->mtu = 9000;
  2. GMAC寄存器中启用巨型帧模式（MTU寄存器设值>1518）
  3. 接收缓冲区增大到对应大小（如9216字节）
  4. PHY需支持巨型帧（无需特殊配置，由MAC处理）

  • 接收帧时，同样捕获接收时间戳

错误标志 寄存器位置 常见原因 排查方法
TX_ERR（发送错误） ISR 半双工冲突超限、PHY未link、TX_CLK缺失 检查PHY链路状态，确认双工模式匹配

RX_FRAME_ERR RDES0 帧长度小于64字节（runt frame） 检查对端设备，可能是噪声导致
MDIO超时 NSR PHY地址错误、PHY未上电、MDC时钟过高 降低MDC频率，确认硬件连接

第十章：常见问题与FAQs

Q1：PHY无法识别/读取ID全是0xFFFF或0x0000

可能原因：

  • 硬件引脚连接错误（MDC/MDIO反接）
  • PHY地址错误（地址0通常为广播地址，应避开）
  • MDC时钟频率过高（建议起始设为2.5MHz/2分频）
  • PHY芯片未上电或复位引脚电平不正确

排查步骤：

  1. 用示波器测量MDC和MDIO信号，确认有通讯波形
  2. 遍历PHY地址1-31，读取PHYID1/PHYID2寄存器
  3. 检查PHY的复位时序：复位后需等待足够时间（典型10-50ms）
  4. 查看PHY数据手册中的默认地址配置（引脚拉高/拉低）

Q2：能接收数据但无法发送/发送中断不触发

检查清单：

  1. 确认发送描述符的OWN位已正确置1
  2. 确认GMAC_NCR中的TXEN位已使能
  3. 检查TX_CLK是否存在（某些SoC需要配置引脚复用功能）
  4. 半双工模式下检查是否有持续的载波冲突（观察COL引脚）
  5. 确认描述符中的FS/LS标志正确设置——只有设置LS的帧才会触发发送完成中断

Q3：接收数据偶尔丢包或CRC错误率偏高

排查方向：

  • 信号完整性：检查RGMII走线是否等长，建议长度差<5mm
  • 时钟相位：尝试切换GMAC内部的TX/RX时钟延迟配置（调试方法见下文）
  • 电源噪声：PHY的模拟电源（AVDD）需使用LDO单独供电
  • 接地：确保PHY的AGND和DGND在单点连接

Q5：MAC地址应该从哪里获取？

推荐方案（按优先级排列）：

  1. 芯片唯一ID：使用MCU的96位唯一ID（如STM32的UID）计算生成
  2. OTP/eFuse：量产时在芯片的一次性可编程区域写入
  3. 外部EEPROM：板载存储MAC地址
  4. 默认值+随机：开发阶段可临时使用默认值，配合随机化避免局域网冲突

Q6：半双工模式下工作不稳定

半双工模式对时序和信号完整性要求更高。建议：

  1. 确认PHY协商结果确实是半双工（读取PHY_ANLPAR寄存器）
  2. 检查冲突计数器是否异常增加
  3. 若非必须，强烈建议改用全双工模式
  4. 如需半双工，减小网络总长度，使用高质量双绞线

Q7：如何实现多GMAC实例（双网口）？

大部分SoC支持2个GMAC实例，要点：

  1. 为每个实例分配独立的描述符环和缓冲区
  2. 使用不同的中断向量（如GMAC0_IRQn和GMAC1_IRQn）
  3. 为每个接口创建独立的struct netif
  4. 确保两个GMAC的PHY地址不同
  5. 注意总线带宽——双千兆全速时，DMA带宽可能成为瓶颈

Q8：复位后需等待多久才能访问PHY？

典型时序要求：

  • PHY上电后：需等待10-50ms（内部模拟电路稳定）
  • PHY软复位后：需等待约100-500μs（数据手册有具体值）
  • 自协商完成：通常1-3秒（最坏可达5秒）
  • 建议在自协商等待期间循环读取BMSR，超时则回退到强制模式
